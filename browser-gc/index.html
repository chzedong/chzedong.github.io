🧠 心智模型

浏览器的 JavaScript 引擎会自动清理「再也访问不到」的对象，但清理过程会暂停主线程。GC 友好的本质是：减少短命对象的创建、避免无意义的长期引用、让垃圾对象在正确的时机被回收。

背景与问题 L1

JavaScript 是自动内存管理的语言，开发者不需要手动 malloc/free。这带来便利，也带来隐形成本：当堆内存增长到一定程度，引擎必须停下来回收垃圾，这个停顿可能导致动画掉帧、交互延迟。

常见问题：

  • 频繁 GC：热路径创建大量短命对象，GC 反复触发。
  • 长时间 GC：老年代对象过多，Full GC 停顿明显。
  • 内存泄漏：对象被长期引用，无法释放，最终崩溃或卡顿。

快速上手 L2

最小可运行示例：对象池

在热路径中复用对象，避免反复创建：

const pool = []
function getParticle() {
  return pool.pop() || { x: 0, y: 0, vx: 0, vy: 0 }
}
function releaseParticle(p) {
  p.x = p.y = p.vx = p.vy = 0
  pool.push(p)
}

// 使用
const p = getParticle()
p.x = 100
// ... 使用完毕
releaseParticle(p)

常见场景

  • 游戏/动画：粒子系统、 bullets、频繁更新的坐标数据。
  • 大数据表格：虚拟滚动 + 对象复用，避免一次性渲染万行 DOM。
  • SPA 路由：离开页面时取消订阅、释放事件监听。

用法详解 L2

GC 友好的操作

操作说明示例
对象复用 / 对象池 热路径中重复利用对象，减少分配和 GC 频率 pool.pop() / pool.push()
使用 WeakMap / WeakSet / WeakRef 不阻止垃圾回收的关联数据存储 new WeakMap()
TypedArray 数值数据用连续内存，减少对象头开销 new Float32Array(1024)
避免闭包捕获大对象 闭包会延长被捕获变量的生命周期 把大数据作为参数传入，而非闭包变量
字符串 Builder 模式 大量字符串拼接时先存入数组再 join parts.join('')
批量 DOM 操作 减少布局抖动和临时对象 DocumentFragment、requestAnimationFrame
取消事件订阅 防止 DOM 与回调长期互相引用 removeEventListener、AbortController

GC 不友好的操作

操作为什么不友好改进方向
热路径大量 {} / [] 每帧都产生短命对象，触发频繁 GC 复用对象、使用 TypedArray
闭包捕获大对象却未释放 大对象生命周期被意外延长 只捕获必要字段、显式置 null
全局缓存无淘汰 内存只增不减，老年代膨胀 LRU、TTL、WeakMap
循环中字符串拼接 产生大量中间字符串对象 数组 join、模板字符串批量生成
setInterval 未清理 回调持有上下文，无法回收 组件卸载时 clearInterval
Detached DOM DOM 已从树移除，仍被 JS 引用 移除 DOM 同时解除事件监听和引用
订阅总线不取消 组件销毁后仍接收消息，持有引用 返回 unsubscribe、使用 Dispose 模式

最佳实践

💡 最佳实践
  • 用 WeakMap 给对象附加元数据，避免干扰对象生命周期。
  • 热路径优先用 const 和局部变量，减少属性访问和对象分配。
  • 大数据用 ArrayBuffer / TypedArray，而不是对象数组。
  • 组件/路由销毁时统一清理 timer、subscription、事件监听。
  • 缓存必须设置容量上限或过期策略。
  • 用 Chrome DevTools 的 Memory 面板定期拍 Heap Snapshot 验证。

内存陷阱

⚠️ 常见陷阱
  • console.log 大对象：DevTools 保留引用，导致 Heap Snapshot 中对象无法释放。
  • 忘记 removeEventListener：DOM 与回调形成循环引用，现代浏览器已能处理，但仍会延长生命周期。
  • Vue/React 中把大对象放进 reactive/ref：递归代理所有属性，增加内存和追踪开销。
  • 图片/Canvas 不释放：大 Image 对象或 OffscreenCanvas 长期持有。
  • 无限滚动不回收 DOM：节点数持续增长，滚动性能崩溃。

V8 分代 GC 如何工作

区域存放对象回收算法特点
新生代（New Space）短命对象Scavenge（复制算法）频繁、停顿短
老生代（Old Space）存活较久的对象Mark-Sweep + Mark-Compact低频、停顿可能长
大对象区（Large Object Space）超过阈值的大对象单独管理不移动，直接释放

现代 V8 还引入了增量标记（Incremental Marking）、并发标记（Concurrent Marking）、并行清理来减少停顿，但主线程仍会被部分阻塞。

为什么会卡顿

  1. 分配过快：新生代很快填满，Scavenge 频繁触发。
  2. 晋升过多：短命对象在两次 GC 间幸存，进入老生代，导致 Full GC 成本上升。
  3. 老年代膨胀：大缓存、泄漏对象使 Mark-Compact 耗时增加。
  4. 内存碎片：长期运行后，Compact 需要移动大量对象。

架构友好/不友好 L4

✅ GC 友好架构

  • 组件生命周期管理：卸载时统一 dispose。
  • 虚拟滚动 / 对象池：复用 DOM 与数据对象。
  • 懒加载 + 按需卸载：路由/模块离开时释放资源。
  • Worker 卸载计算：大数据处理放到 Worker，避免阻塞主线程 GC。
  • 有界缓存 + LRU：缓存不会无限增长。
  • 细粒度响应式：只追踪必要依赖，减少派生对象。
  • 资源注册表：集中管理图片、音频、WebGL 纹理。

❌ GC 不友好架构

  • 全局单例大仓库：所有状态长期存活，老年代膨胀。
  • 无限制事件总线：订阅只增不减，组件销毁仍被引用。
  • 无限列表不回收：DOM 节点和数据无限累积。
  • 全量快照状态管理：每次更新复制整个状态树。
  • 长生命周期闭包：回调捕获大量上下文。
  • 图片/Canvas 无释放：纹理和位图长期占用 GPU/内存。

问题：组件频繁创建销毁，如何避免内存泄漏和 GC 压力？

直觉方案：每个组件自己管理事件监听，卸载时手动清理。

实际做法：框架提供统一的 Effect / Scope 系统，订阅与组件生命周期绑定；同时用 WeakMap 存储内部元数据，不干扰用户对象。

差异分析：手动清理容易遗漏，且难以处理异步副作用；生命周期绑定 + 自动追踪能系统性防止泄漏，并减少长期引用。

GC vs 手动内存管理 L3

维度自动 GC（JavaScript）手动管理（C/C++/Rust）所有权 + GC 混合（Swift）
开发效率高低中
运行时停顿存在 GC pause无极小
内存泄漏风险仍存在（逻辑泄漏）Use-after-free / 泄漏循环引用需处理
可预测性低高高
适用场景Web、业务应用系统、游戏引擎移动应用

历史演进 L4

时间事件意义
1959Lisp 引入自动 GC首次在高级语言中实现自动内存管理
1960s引用计数普及简单直接，但无法处理循环引用
1984分代 GC 提出利用「大多数对象短命」的假设优化回收效率
2008V8 发布引入高效分代 GC，推动 JS 高性能化
2018+V8 并发/增量标记显著降低 Full GC 造成的页面卡顿
2021+WeakRef / FinalizationRegistryJS 获得更细粒度的生命周期控制能力

学习要点总结 L1

  • GC 友好的核心是：少创建短命对象、避免无意义长期引用、及时释放资源。
  • 热路径用对象池 / TypedArray，避免每帧分配对象。
  • 用 WeakMap / WeakSet / WeakRef 存储不干预生命周期的关联数据。
  • 组件/路由/模块销毁时统一清理 timer、subscription、事件监听、DOM 引用。
  • 缓存必须有容量上限或过期策略，避免老年代膨胀。
  • 架构上采用生命周期绑定、虚拟滚动、Worker 卸载、有界缓存等模式。
