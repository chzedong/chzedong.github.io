解构 Immer — Proxy draft → 不可变

Create the next immutable state by mutating the current one · v10

28,945 ★MITMichel Weststrate

快速上手

最小示例

import { produce } from 'immer'

const baseState = [
  { title: 'Learn Immer', done: true },
  { title: 'Use Immer', done: false },
]

const nextState = produce(baseState, draft => {
  draft[1].done = true       // 直接修改 proxy
  draft.push({ title: 'Tweet about it', done: false })
})

// baseState 不变！nextState 是新对象

Redux reducer 场景

// ❌ 传统写法
const reducer = (state, action) => {
  switch (action.type) {
    case 'ADD_TODO':
      return { ...state, todos: [...state.todos, action.payload] }
  }
}

// ✅ Immer 写法 — 直接"修改"
const reducer = produce((draft, action) => {
  switch (action.type) {
    case 'ADD_TODO':
      draft.todos.push(action.payload)
  }
})

1. 项目背景

解决了什么问题

JavaScript 的不可变更新是出了名的啰嗦。深层嵌套对象的展开会让你怀疑人生。

Immer 的思路：让你用可变语法写不可变逻辑。给原始对象包一层 Proxy，在 Proxy 上直接用 draft.name = 'new' 修改，Immer 在背后创建浅拷贝链，最终返回一个新对象，原始对象毫发无伤。

🧠 如果让你来实现

问题：用 Proxy 实现"可变语法 → 不可变结果"，怎么追踪哪些字段被改过？

直觉方案：Proxy 的 set trap 里每次修改同步写回一个新对象，一层一层浅拷贝。

Immer 的做法：写时不立即拷贝——只在第一次修改时调用 prepareCopy() 做浅拷贝。之后所有修改打在 copy_ 上，base_ 保持不动。最终 finalize() 时只拷贝修改过的层级，没改的分支直接复用 base_ 的引用。

差异分析：直觉方案每个 set 都做一次浅拷贝，但写一个深层字段时要为整条路径上的每个父节点都做浅拷贝——产生大量中间对象。Immer 把拷贝和修改分离成两个阶段（draft 阶段→finalize 阶段），批量处理，避免了中间拷贝。

草稿状态结构

src/core/proxy.ts — ProxyObjectState
interface ProxyObjectState {
  type_: ArchType.Object
  base_: any            // ★ 原始对象（不变）
  copy_: any            // ★ 浅拷贝（所有修改打在这里）
  draft_: Drafted       // ★ Proxy 实例（给用户操作的）
  modified_: boolean    // 是否修改过
  finalized_: boolean   // 是否已终局化
  assigned_: Map        // 记录了哪些属性被修改/删除
  parent_: ImmerState   // 父 draft（用于递归标记 modified_）
  scope_: ImmerScope    // 所属 scope
}
💡 设计意图：base_ 和 copy_ 分离是关键。修改打 copy_ 上，未修改的字段直接读 base_。finalize 时如果 modified_ == false，直接返回 base_（零拷贝）。

Proxy 陷阱：get — 惰性拷贝

src/core/proxy.ts — get trap（精简）
get(state, prop) {
  const source = latest(state)   // copy_ ?? base_
  const value = source[prop]
  if (state.finalized_ || !isDraftable(value)) return value

  // 检查是否是新值（不是 base_ 上的原始值）
  if (value === peek(state.base_, prop)) {
    prepareCopy(state)           // ★ 第一次访问时才创建 copy_！
    return state.copy_[prop] = createProxy(scope, value, state)
  }
  return value                   // 已经 draft 过的直接返回
}
💡 关键设计：peek() 检查值是否等于 base_ 上的原始值。只有当用户真的要修改时 prepareCopy() 才触发浅拷贝。大结构只读操作零开销。

Proxy 陷阱：set — 智能跳过

src/core/proxy.ts — set trap（精简）
set(state, prop, value) {
  if (!state.modified_) {
    const current = peek(latest(state), prop)
    if (is(value, current) && (value !== undefined || has(state.base_, prop)))
      return true               // ★ 值没变，跳过

    prepareCopy(state)          // 第一次修改才浅拷贝
    markChanged(state)          // 标记 modified_，递归冒泡到 parent_
  }
  state.copy_[prop] = value
  state.assigned_.set(prop, true)
  handleCrossReference(state, prop, value)
  return true
}
💡 设计意图：is(value, current) 用的是结构性相等（不是 ===），所以 draft.x = {a:1}; draft.x = {a:1} 不会触发两次拷贝。markChanged 递归冒泡到所有父 draft。

终局化：finalize

src/core/finalize.ts — finalize()
function finalize(rootScope, value) {
  if (isFrozen(value)) return value
  const state = value[DRAFT_STATE]
  if (!state) return handleValue(value, handledSet, rootScope)

  if (!state.modified_) return state.base_  // ★ 未修改，直接返回原对象

  if (!state.finalized_) {
    const { callbacks_ } = state            // 执行所有子 draft 的回调
    while (callbacks_.length > 0) {
      callbacks_.pop()!(rootScope)
    }
    generatePatchesAndFinalize(state, rootScope)
  }
  return state.copy_                        // 返回复制品
}
💡 设计意图：callbacks_ 是子 draft 注册的回调——父 draft 终局化时，子 draft 将自身的 copy_ 写回到父的 copy_ 中。保证深度嵌套对象的最终结果只包含修改过的路径上的新对象。

5. 架构设计

设计模式

模式位置作用
Proxy 代理Proxy.revocable(target, traps)拦截读写，实现 copy-on-write
写时复制prepareCopy() 惰性触发只在修改时做浅拷贝
回调链表state.callbacks_子 draft 终局化时通知父 draft 更新引用
Scope 隔离enterScope / leaveScope隔离一次 produce 调用的所有 draft
🧠 为什么用 Proxy.revocable？

直觉方案：创建普通 Proxy，等 GC 自动回收。

Immer 的做法：Proxy.revocable(target, traps) 返回 {proxy, revoke}，在 revokeScope() 时显式调用 revoke() 断绝 proxy→target 关联。断绝后任何对 proxy 的操作都抛 TypeError。

差异分析：防止用户在 produce 回调结束后继续持有 draft（常见错误），revoke 让这种误用立即报错，而不是静默产生不可预期的行为。

draft 终局化回调链

sequenceDiagram
    participant User as 用户
    participant Proxy as Draft Proxy
    participant State as ImmerState
    participant Parent as Parent Draft
    participant Scope as ImmerScope

    User->>Proxy: draft.child.name = 'x'
    Proxy->>State: prepareCopy + markChanged
    State->>Parent: markChanged(parent_) 递归冒泡
    User->>Scope: recipe 执行完毕
    Scope->>State: processResult → finalize
    State->>State: 执行 callbacks_[0]
    State->>Parent: updateDraftInParent(parent, draft, finalized)
    Parent->>Parent: parent.copy_[key] = finalizedValue
    Scope->>User: 返回 copy_ 树（已冻结）

图：嵌套 draft 的终局化回调链

⚠️ 易错点

错误用法：在 produce 回调外用 draft 引用。

let draftRef
const next = produce(state, draft => {
  draftRef = draft  // ❌ 存储 draft 引用
  draft.a = 1
})
draftRef.b = 2       // ❌ 报错：proxy 已 revoke

原因：revokeScope() 在 produce 返回后立即调用 proxy.revoke()，此后任何对 draft 的操作都抛 TypeError。

7. 学习要点

可复用的工程实践

  1. 惰性浅拷贝：第一次写时触发 prepareCopy()，大结构只读操作零开销
  2. 回调链表管理子树：state.callbacks_ 统一在 finalize 时批量执行
  3. Proxy.revocable 防误用：而不是靠文档提醒"别在回调外用 draft"
  4. Scope 隔离：嵌套 produce 调用互不干扰

可借鉴的架构模式

  1. 写时复制 + 批量终局化：分离修改收集和结果生成，消除中间拷贝
  2. 结构性相等判断：is() 深比较防止无效修改触发拷贝
  3. updateDraftInParent：支持同一 draft 出现在多个父节点中
