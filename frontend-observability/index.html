前端可观测性实战

告警与埋点的方案、原理与架构思考：让前端系统从「黑盒」变成「可观测」。

Concept L1-L4 Sentry · OpenTelemetry · Web Vitals
L1 概览 L2 用法 L3 原理 L4 精通

🧠 心智模型

前端可观测性 = 日志（Logs）+ 指标（Metrics）+ 链路（Traces）。告警是「发现问题」的手段，埋点是「理解行为」的手段；两者结合，既能知道系统坏了，也能知道用户怎么用的。

背景与问题 L1

前端代码运行在用户浏览器里，环境复杂、网络不可控、错误难以复现。传统后端监控无法覆盖前端特有的问题：

  • 错误感知滞后：用户截图反馈时，已经影响大量人。
  • 性能黑盒：首屏慢、交互卡顿，但不知道瓶颈在哪。
  • 行为不可见：用户点了哪里、在哪个步骤流失，只能靠猜。
  • 多端差异：浏览器、小程序、桌面端，问题表现各不相同。

告警解决「系统有没有病」的问题，埋点解决「用户怎么用」的问题。好的架构师会把二者融入研发流程，而不是事后补丁。

快速上手 L2

最小可运行示例：Sentry 接入

import * as Sentry from '@sentry/browser'

Sentry.init({
  dsn: 'https://xxx@xxx.ingest.sentry.io/xxx',
  integrations: [Sentry.browserTracingIntegration()],
  tracesSampleRate: 0.1,
  replaysSessionSampleRate: 0.01,
})

// 自动捕获未处理异常
throw new Error('something went wrong')

// 手动上报
Sentry.captureMessage('User reached payment page')

最小可运行示例：自定义埋点

function track(event, props = {}) {
  const payload = {
    event,
    props,
    ts: Date.now(),
    url: location.href,
    ua: navigator.userAgent,
  }
  navigator.sendBeacon('/api/track', JSON.stringify(payload))
}

track('button_click', { id: 'submit-order' })

最佳实践

💡 最佳实践
  • 错误去重与聚合：相同错误合并，避免告警风暴。
  • 采样率分层：错误 100% 采集，性能 10%，全埋点 1% 或按用户采样。
  • 关联上下文：每条错误/事件带上 release 版本、用户 ID、页面路径、设备信息。
  • 埋点文档化：维护埋点字典，定义事件名、属性、触发时机。
  • 优先级分级：P0 错误立即通知，P1 次日汇总，P2 周报回顾。
  • 隐私合规：敏感字段脱敏，符合 GDPR/个人信息保护法要求。

常见陷阱

⚠️ 常见陷阱
  • 告警疲劳：阈值过低，导致群里告警被忽略。
  • 全量采集：不采样，数据量和成本爆炸。
  • 忽视用户隐私：采集设备 ID、手机号等敏感信息。
  • 只看不行动：埋点数据存在 BI 平台，但没有反馈到产品迭代。
  • 前后端口径不一致：同一个指标，前端埋点和服务端统计对不上。

图：可观测性三角与告警、埋点的关系

告警链路

  1. 采集：SDK 捕获异常、性能指标、业务事件。
  2. 上报：通过 sendBeacon、XHR、WebSocket 发送到服务端。
  3. 清洗：去重、格式化、补全上下文。
  4. 聚合：按错误指纹、时间段、版本维度汇总。
  5. 告警：触发阈值时通过 PagerDuty、钉钉、飞书、邮件通知。
  6. 恢复：错误率回落后自动关闭告警，并记录处理过程。

埋点数据流

  1. 触发：用户行为或系统事件产生。
  2. 采集：SDK 记录事件名、属性、时间、上下文。
  3. 缓冲：本地队列，批量上报，失败重试。
  4. 传输：压缩后发送到埋点网关。
  5. 清洗：校验、补全、去重、格式转换。
  6. 存储：进入数据仓库或实时数仓。
  7. 分析：漏斗、留存、路径、归因。

架构思考 L4

好的架构师如何利用告警与埋点

✅ 正确姿势

  • 把监控纳入 Definition of Done，新功能上线必带监控。
  • 告警分层： On-Call 处理 P0，团队日会看 P1，周报回顾 P2。
  • 埋点驱动产品：A/B 测试、漏斗分析、留存追踪。
  • 建立可观测性基线：知道「正常」长什么样，才能识别异常。
  • 错误与发布关联：每个错误都知道是哪个版本引入的。
  • 成本意识：采样、聚合、冷热分层存储。

❌ 常见反模式

  • 监控只是运维的事，开发不管。
  • 所有错误都告警，导致告警疲劳。
  • 埋点无限扩张，采集一切但不分析。
  • 线上出问题才加监控，平时不维护。
  • 前后端监控割裂，无法串联完整链路。
🧠 如果让你来设计一套前端可观测体系...

问题：如何在有限的成本下，既快速发现线上问题，又能理解用户行为？

直觉方案：接入 Sentry 捕获错误，再用一个埋点 SDK 采集所有点击。

更好的做法：把告警和埋点统一到一个数据平台，错误事件本身也是一种埋点；通过 Trace ID 把前端请求、后端调用、数据库查询串起来；用采样和分级降低成本。

差异分析：分离的体系会导致数据孤岛，排查问题时需要切换多个平台；统一的 Trace + 事件模型能缩短 MTTR（平均修复时间），并让行为数据与系统数据相互印证。

学习要点总结 L1

  • 前端可观测性 = 日志 + 指标 + 链路，告警和埋点是两个核心抓手。
  • 告警用于发现系统问题，分为错误、性能、业务、可用性四类。
  • 埋点用于理解用户行为，分为代码埋点、可视化埋点、无埋点、服务端埋点。
  • 好的架构师会把监控纳入研发流程，分层告警、合理采样、关联上下文。
  • 推荐组合：Sentry / OpenTelemetry 做采集，Grafana / ClickHouse / 自建平台做分析。
  • 避免告警疲劳、过度采集、隐私违规、前后端口径不一致。
