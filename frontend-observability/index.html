🧠 心智模型

前端可观测性不是"接入 Sentry"或"埋几个点"，而是 Logs + Metrics + Traces + RUM + 埋点 的体系化工程。错误监控回答"系统有没有病"，性能监控回答"哪里慢"，埋点回答"用户怎么用"，告警是让这些数据产生行动的手段。

背景与问题 L1

前端代码运行在用户浏览器里，环境复杂、网络不可控、错误难以复现。传统后端监控无法覆盖前端特有的问题：

  • 错误感知滞后：用户截图反馈时，问题已经影响大量人；没有 stack trace 和用户环境信息，本地难以复现。
  • 性能黑盒：首屏慢、交互卡顿，但不知道瓶颈在 DNS、TTFB、JS 执行、渲染还是第三方脚本。
  • 行为不可见：用户点了哪里、在哪个步骤流失、A/B 实验效果如何，只能靠猜。
  • 多端差异：浏览器、小程序、桌面端、不同网络条件下问题表现各不相同。

后端监控主要关注服务可用性和接口延迟，而前端监控必须面对 不可控的运行时环境、异步事件驱动、SPA 路由切换、第三方脚本干扰 等独特挑战。

快速上手 L2

示例 1：原生错误 + 性能采集 SDK

不依赖第三方库，直接用浏览器 API 搭建最小可运行采集器：

最小可运行采集器 MDN: PerformanceObserver 
// 错误采集
window.addEventListener('error', (event) => {
  report({ type: 'error', message: event.message, filename: event.filename, lineno: event.lineno })
})
window.addEventListener('unhandledrejection', (event) => {
  report({ type: 'unhandledrejection', reason: String(event.reason) })
})

// 性能采集：LCP、CLS、INP
const observer = new PerformanceObserver((list) => {
  for (const entry of list.getEntries()) {
    if (entry.entryType === 'largest-contentful-paint') {
      report({ type: 'lcp', value: entry.startTime, element: entry.element?.tagName })
    }
    if (entry.entryType === 'layout-shift' && !entry.hadRecentInput) {
      report({ type: 'cls', value: entry.value })
    }
    if (entry.entryType === 'event' && entry.interactionId) {
      report({ type: 'inp', value: entry.duration, interactionId: entry.interactionId })
    }
  }
})
observer.observe({ type: 'largest-contentful-paint', buffered: true })
observer.observe({ type: 'layout-shift', buffered: true })
observer.observe({ type: 'event', buffered: true, durationThreshold: 0 })
关键参数：buffered: true 允许获取在观察者注册前已经产生的性能条目，这对 SDK 延迟加载很重要。durationThreshold: 0 是采集 INP 的前提，否则 Event Timing 条目可能被过滤。

示例 2：生产级 track() 函数

一个带队列、批量、失败重试、页面关闭兜底的埋点 SDK 最小实现：

生产级埋点 SDK 最小实现 W3C Beacon API 
function createTracker(endpoint, { sampleRate = 1.0, batchSize = 10, flushInterval = 5000 } = {}) {
  const queue = []
  let timer = null

  function send(events) {
    if (Math.random() > sampleRate) return // 采样
    const payload = JSON.stringify(events)
    const blob = new Blob([payload], { type: 'application/json' })

    // 优先 sendBeacon，页面关闭时也能尽力发送
    if (navigator.sendBeacon && navigator.sendBeacon(endpoint, blob)) return

    // 兜底：fetch keepalive
    fetch(endpoint, { method: 'POST', body: blob, keepalive: true }).catch(() => {
      // 失败时存入 localStorage/IndexedDB 稍后重试
      saveToRetryStore(events)
    })
  }

  function flush() {
    if (queue.length === 0) return
    const batch = queue.splice(0, queue.length)
    send(batch)
  }

  function track(event, props = {}) {
    queue.push({
      event,
      props,
      ts: Date.now(),
      url: location.href,
      sessionId: getSessionId(),
      eventId: generateEventId(),
    })
    if (queue.length >= batchSize) flush()
    if (!timer) timer = setInterval(flush, flushInterval)
  }

  // 页面隐藏时强制 flush
  document.addEventListener('visibilitychange', () => {
    if (document.visibilityState === 'hidden') flush()
  })

  return { track, flush }
}
设计要点：1) eventId 用于服务端去重；2) sessionId 用于漏斗分析和会话一致性采样；3) visibilitychange 比 beforeunload 更可靠，尤其在移动端；4) sendBeacon 返回 false 说明超出浏览器队列配额，需要降级。

用法详解 L2

错误监控

前端错误监控的核心是捕获三类信息：JavaScript 运行时异常、Promise 未捕获拒绝、资源加载失败。浏览器提供了不同 API，但各自有覆盖盲区。

API捕获什么盲区锚点
window.onerrorJS 运行时异常无法捕获资源加载失败、Promise 错误HTML Standard §8.1.3.10
window.addEventListener('error')JS 异常 + 资源加载失败Promise 拒绝DOM Standard
unhandledrejection未处理的 Promise 拒绝同步异常HTML Standard
React ErrorBoundaryReact 组件渲染错误事件处理、异步代码、SSR 错误React 文档
⚠️ 踩坑：跨域脚本错误

加载第三方 CDN 脚本时，如果脚本没有 crossorigin="anonymous" 且 CDN 没有返回 Access-Control-Allow-Origin，错误信息会变成 Script error.，没有文件名、行号、堆栈。

调试方法：在测试环境强制要求所有外部脚本开启 CORS，并在 Sentry 中查看错误样本是否包含完整 stack。

错误指纹是错误聚合的关键。一个朴素的做法是把错误消息直接作为 key，但不同用户遇到同一个 bug 时消息可能包含变量值（如 Cannot read property 'foo' of undefined 对比 Cannot read property 'bar' of undefined）。成熟的做法是：

  1. 解析 stack trace，提取函数名和文件路径；
  2. 对消息做正则化，移除 UUID、数字、用户 ID 等变量；
  3. 结合 release 版本和构建产物 sourcemap 还原原始代码位置；
  4. 生成稳定的 fingerprint 用于聚合。

性能监控（Web Vitals + RUM）

前端性能监控依赖 W3C Performance Timeline 系列 API。核心入口是 PerformanceObserver，它取代了旧的轮询 performance.getEntries() 方式。

指标含义Good / Needs Improvement / Poor对应 API
LCP最大内容元素渲染时间≤2.5s / ≤4s / >4slargest-contentful-paint
INP交互到下一帧绘制最大延迟≤200ms / ≤500ms / >500msevent (Event Timing)
CLS累积布局偏移≤0.1 / ≤0.25 / >0.25layout-shift
TTFB首字节时间≤800ms / ≤1800ms / >1800msnavigation
FCP首次内容绘制≤1.8s / ≤3s / >3spaint
💡 为什么是这些指标和阈值？
  • LCP：用户感知"页面主要内容已加载"的关键时刻。阈值 2.5s 来自 Google 对真实用户跳出率和体验的研究（CrUX 数据），是 75th percentile 的 good 线。
  • INP：2024 年 3 月替代 FID，因为 FID 只测首次交互且多数网站轻松达标，无法反映整体交互卡顿。INP 取整个会话中最差交互延迟，更能代表真实体验。
  • CLS：量化"页面加载时元素突然跳动"的体验。分数 = impact fraction × distance fraction，0.1 以下用户基本无感知。
  • 75th percentile：Google 评估标准是 75% 的访问达到 good。这比中位数更严格，但比 95th percentile 更稳定，避免少数极端网络环境导致整体失败。

SPA 路由切换的性能陷阱：标准 Navigation Timing 只测量完整页面加载，不捕获 history.pushState 引起的软导航。因此 SPA 需要手动测量：

SPA 路由切换手动测量
let routeStartTime = performance.now()
window.addEventListener('popstate', () => { routeStartTime = performance.now() })

// 在路由切换完成、关键元素渲染后调用
function markRouteComplete(routeName) {
  const duration = performance.now() - routeStartTime
  report({ type: 'spa_route', route: routeName, duration })
}
不同框架的"路由完成"定义不同：React Router 的 useEffect、Vue Router 的 afterEach、Next.js 的 router.events。必须约定一个统一标准，否则跨页面数据不可比。

埋点方案

埋点不是简单地在按钮上加 onClick，而是一套事件模型、采集机制、治理流程的体系。

方案实现方式优点缺点适用场景
代码埋点开发手动调用 track()精确、可控、属性完整侵入代码、维护成本高、容易遗漏核心业务流程
可视化埋点运营圈选页面元素配置事件非侵入、业务人员可操作只能覆盖静态元素、动态内容难适配活动页、运营位
无埋点/全埋点自动采集点击、曝光、页面切换覆盖全面、回溯能力强数据量大、噪音多、隐私合规风险探索性分析
服务端埋点BFF/API 网关记录业务行为防篡改、数据一致性高无法捕获前端交互细节关键业务事件

采样策略

采样不是随意定一个百分比，而是要在成本、准确性、业务需求之间做权衡。

策略原理优点缺点适用
随机采样每个事件独立按概率丢弃实现简单破坏会话/漏斗一致性性能指标、日志
会话哈希采样按 session_id 哈希取模同一会话内数据一致无法按事件重要性调整用户行为分析
动态采样根据错误率/流量实时调整异常时提高覆盖率实现复杂，需避免雪崩错误监控
服务端采样全量上报后服务端过滤策略灵活、可回溯带宽和客户端开销大小规模或关键业务
⚠️ 失败场景：采样破坏漏斗分析

如果"进入商品页"和"提交订单"采用独立随机采样，可能出现商品页事件被采样而订单事件未被采样的情况，导致转化率失真。

解决方法：漏斗相关事件使用会话级一致性采样；或在服务端保留完整漏斗路径，仅对非漏斗事件采样。

经验采样率不是铁律："错误 100%、性能 10%、全埋点 1%"只是起点。实际应该根据：

  • 错误量：小业务错误少，可以 100%；大业务错误多，可以按指纹/hash 采样；
  • 性能指标的统计需求：估算 p95/p99 需要的样本量；
  • 成本预算：存储、计算、带宽；
  • 合规要求：用户隐私法可能限制可采集的数据量。

Logs、Metrics、Traces 的取舍：

  • Metrics：适合聚合趋势和告警，成本低，但丢失个体细节。
  • Logs：适合保留事件上下文，成本高，检索慢。
  • Traces：适合跨服务调用链分析，前端 Trace 通过 W3C traceparent 与后端串联。

RUM（Real User Monitoring）与合成监控的区别：RUM 采集真实用户的设备、网络、地理位置分布，能发现实验室环境测不出的长尾问题；合成监控在固定环境定期执行，适合回归基线对比。

性能指标采集原理

PerformanceObserver 的工作方式：

  1. 浏览器在内部维护一个性能条目缓冲区（buffer）；
  2. PerformanceObserver 注册后，新产生的条目会推送给回调；
  3. buffered: true 允许一次性获取缓冲区中已有的条目，解决 SDK 延迟加载导致的数据丢失问题；
  4. 不同 entryType 的 buffer 大小有限制，超过后旧条目会被丢弃。

LCP 元素判定：LCP 候选元素包括 <img>、<video> 的海报、block-level 的文本节点等。当用户与页面发生交互（点击、滚动、按键）时，LCP 停止记录，因为后续内容可能是用户主动触发的，不再代表"初始加载体验"。

INP 计算：INP 基于 Event Timing API。对每个交互事件，浏览器测量从事件派发到下一帧绘制的时间，取整个会话中的最大延迟（p98）。这也是为什么长任务阻塞主线程会严重拉高 INP。

CLS 计算：layout shift score = impact fraction × distance fraction。只有视口内可见元素的移动才计入，且用户交互后 500ms 内的移动会被排除（hadRecentInput）。

上报通道的 trade-off：

通道优点限制适用场景
navigator.sendBeacon页面关闭后仍可发送，不阻塞导航64KB 总 quota，无法自定义 headers，无响应页面卸载兜底、心跳
fetch keepalive可自定义 headers/body，更灵活同样有 keepalive quota 限制常规批量上报
同步 XHR页面关闭前可强制发送阻塞页面，体验差，已被弃用最后 fallback
1×1 image beacon跨域简单，兼容性好GET 长度受限，数据量小简单计数、广告追踪
⚠️ 失败场景：sendBeacon 的 64KB 陷阱

Chromium 和 Safari 对 sendBeacon 有总 quota 限制（约 64KB 未发送数据）。如果批量事件过大或失败重试队列积压，sendBeacon 会返回 false，事件静默丢失。

调试方法：监控 sendBeacon 返回值；在实验室环境构造大 payload 测试；使用 fetch keepalive 作为降级。

采样算法

客户端随机采样最简单，但会破坏会话一致性：

随机采样
function shouldSample(sampleRate) {
  return Math.random() < sampleRate
}

会话哈希采样保证同一用户要么全采要么全不采：

会话一致性采样
function hashString(str) {
  let hash = 0
  for (let i = 0; i < str.length; i++) {
    hash = ((hash << 5) - hash) + str.charCodeAt(i)
    hash |= 0
  }
  return Math.abs(hash)
}

function shouldSampleBySession(sessionId, sampleRate) {
  const bucket = hashString(sessionId) % 10000
  return bucket < sampleRate * 10000
}
使用 session_id 作为哈希输入，确保同一会话内所有事件采样结果一致。采样率调整时，建议重新哈希或用两层采样避免样本突变。

动态采样：在错误率升高或流量突增时自动提高采样率。实现时需要注意避免正反馈循环——采样率提高导致上报量增加，进一步触发采样率提高。

图：前端可观测 SDK 的分层架构

关键配置项：

  • sampleRate：全局采样率，可按事件类型覆盖；
  • endpoint：上报域名，建议独立域名避免被业务 CORS/广告拦截影响；
  • batchSize / flushInterval：批量阈值，太小增加请求数，太大数据易丢；
  • privacyRules：字段脱敏规则，如手机号、身份证号、密码框输入；
  • enableSPA：是否启用软导航测量；
  • release：版本号，用于错误与发布的关联。

队列、批量与重试

批量上报的阈值设计是平衡实时性和开销的关键：

  • 数量阈值：如 10 条事件立即发送，适合高频交互场景；
  • 时间阈值：如 5 秒强制 flush，避免事件在队列中停留过久；
  • 页面生命周期阈值：visibilitychange === 'hidden' 时立即 flush。

重试策略：

  1. 首次失败：指数退避重试（1s, 2s, 4s, 8s...）；
  2. 多次失败：转存到 IndexedDB，下次初始化时恢复；
  3. 避免重试风暴：设置最大重试次数和全局并发限制。

数据清洗与关联

Trace 上下文关联：前端 fetch 请求注入 W3C traceparent header，后端接力，实现全链路追踪。需要注意 CORS 预检：

fetch 注入 traceparent
const traceparent = `00-${traceId}-${spanId}-01`
fetch('/api/order', {
  headers: { traceparent }
})
服务端必须在 Access-Control-Allow-Headers 中允许 traceparent，否则跨域请求会被浏览器拦截。这是生产环境最容易遗漏的配置。

数据对账：前端埋点与后端日志、业务数据库经常出现口径不一致。常见原因包括：

  • 网络失败导致事件未上报；
  • adblock / 隐私拦截导致 SDK 未加载；
  • 预渲染/爬虫触发前端事件但无后端动作；
  • 时区/时间戳不一致；
  • 前后端对"一次成功"定义不同。

调试方法：建立端到端对账机制，如"后端订单创建数 ≈ 前端 order_submit 事件数 × (1 - 拦截率) × 上报成功率"。

成本模型

前端可观测性的成本主要来自：

  1. 客户端开销：SDK 体积、CPU 用于序列化和压缩、额外网络请求；
  2. 传输成本：带宽，尤其是全埋点场景；
  3. 存储成本：日志/事件通常是最大头；
  4. 计算成本：清洗、聚合、告警、分析。

估算公式：

日事件量 = DAU × 人均事件数
日存储量(GB) = 日事件量 × 平均事件大小(B) / 1024 / 1024 / 1024
月存储量 = 日存储量 × 保留天数

降本手段：采样、字段裁剪、客户端预聚合、冷热分层、设置合理的保留策略。

隐私与合规

前端埋点最容易触碰隐私红线。必须遵守的原则：

  • 默认不采集 PII（个人可识别信息）；
  • 敏感输入框（密码、身份证、银行卡）必须脱敏或排除；
  • 提供用户同意管理（consent management），未同意前禁止采集；
  • 欧盟 GDPR、中国个人信息保护法要求数据本地化或明确告知用途；
  • 采样和匿名化可以降低合规风险。

如果我来写 L4

🧠 问题 1：错误上报怎么保证不丢？

直觉方案：每次错误发生立即用 fetch 上报。

实际方案：本地队列 + 批量 + sendBeacon 兜底 + IndexedDB 重试。

差异分析：立即 fetch 在页面卸载时会被取消，导致关键错误丢失；批量可以减少请求数；sendBeacon 在页面关闭时仍能尽力发送；IndexedDB 解决离线/网络抖动场景。

反向场景：如果业务对实时性要求极高（如支付失败立即告警），可以关键错误单独通道实时上报，普通错误走批量队列。

🧠 问题 2：性能指标怎么采集？

直觉方案：用 setInterval 轮询 performance.timing。

实际方案：PerformanceObserver + buffered: true。

差异分析：轮询浪费 CPU，且容易错过短暂出现的性能条目；PerformanceObserver 是事件驱动，只在有新条目时触发，且能获取已缓冲的历史条目。buffered: true 解决了 SDK 延迟加载导致的数据丢失。

反向场景：如果需要兼容 IE11，只能回退到轮询 performance.getEntriesByType。

🧠 问题 3：埋点怎么实现？

直觉方案：每个按钮手动加 onClick={() => track('click')}。

实际方案：全局事件代理 + 圈选配置/埋点字典。

差异分析：手动埋点容易遗漏、命名不一致、维护成本高。全局代理配合 data-track-id 等标记，可以在不修改业务代码的情况下新增埋点；代码埋点和可视化埋点结合是大型业务的常见做法。

反向场景：核心交易链路（如提交订单）建议保留代码埋点，因为全局代理可能因 DOM 变化或事件冒泡被阻止而失效。

🧠 关键决策：为什么 INP 替代 FID？

Google 数据显示 93% 的网站已经能轻松通过 FID（<100ms），因为 FID 只测量首次交互的输入延迟，不反映后续交互的卡顿。INP 测量整个会话中所有交互到下一帧绘制的最大延迟，更能代表真实用户体验。这个变更说明性能指标必须从"能过关"转向"真实体验"。

学习要点总结 L1

可复用的工程实践

  1. 错误监控三件套：error 事件 + unhandledrejection + ErrorBoundary，缺一不可。
  2. 性能采集用 PerformanceObserver：避免轮询，记得开启 buffered: true。
  3. 埋点必须有 eventId 和 sessionId：前者用于去重，后者用于会话分析和一致性采样。
  4. 上报通道分层：正常用 fetch，页面隐藏用 sendBeacon，失败转存 IndexedDB。
  5. 采样要考虑漏斗一致性：会话哈希采样比纯随机采样更适合用户行为分析。
  6. 隐私默认最小化：PII 不采集，敏感输入脱敏，consent 前置。

可借鉴的架构模式

  1. 采集/处理/传输/存储分层：每层的故障域独立，便于替换和降级。
  2. 客户端预聚合：如 LCP 只需要最终值，不需要上报所有中间候选。
  3. Trace 上下文传播：前端注入 traceparent，串联后端调用链。
  4. 冷热分层存储：热数据用于实时告警，冷数据用于离线分析。
