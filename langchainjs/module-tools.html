🔧 Tools & MCP

StructuredTool、工具调用协议、MCP 适配器、多 Provider 工具生态。

扩展模块@langchain/core/toolsMCP 支持

1. 模块定位

Tools 系统是 LangChain Agent 的"手和脚"——它让 LLM 能够调用外部函数、查询数据库、搜索网络。Tools 系统设计为一个独立的子包 @langchain/core/tools，同时提供 MCP（Model Context Protocol）适配器来对接外部工具服务器。

2. StructuredTool

StructuredTool 是所有工具的基类。它接受 Zod schema 来定义输入参数的结构，并提供 invoke() 方法执行工具逻辑：

libs/langchain-core/src/tools/types.ts:58+
// ★ Tool 定义了 schema（输入类型）+ func（业务逻辑）+ metadata
export abstract class BaseTool {
  abstract name: string
  abstract description: string
  abstract schema: InteropZodObject  // ★ Zod v3/v4 兼容

  abstract _call(
    input: InferInteropZodInput<this['schema']>,
    runManager?: CallbackManagerForToolRun,
    config?: RunnableConfig
  ): Promise<string>
}
  • StructuredTool — 标准工具：schema + func
  • DynamicTool — 动态工具：运行时解析 schema
  • tool() helper — 工厂函数，一行创建工具
  • RunnableToolLike — 任何实现 Runnable 的都可以作为工具

3. 工具调用协议

LangChain 的工具调用遵循标准协议：

  1. 用户消息 → Model 产生 AIMessage（含 tool_calls 字段）
  2. tool_calls 包含 { name, args, id } — 标准化的工具调用描述
  3. Agent 解析 tool_calls → 执行对应工具 → 返回 ToolMessage
  4. ToolMessage 追加到对话历史，继续下一次推理循环

4. MCP 适配器

@langchain/mcp-adapters 将 MCP（Model Context Protocol）服务器的工具转换为 LangChain 的 Tool 对象：

  • MCP Tools — MCP 服务器暴露的工具函数 → LangChain StructuredTool
  • MCP Resources — MCP 资源（文件、数据库等）→ LangChain 可查询的数据源
  • MCP Prompts — MCP 提示模板 → LangChain PromptTemplate

这使得任何兼容 MCP 的工具服务器（如数据库客户端、API 网关）都可以被 LangChain Agent 直接调用。

5. "如果我来写" 对比思考

🧠 问题：如何让工具定义与类型系统对齐？

直觉方案：JSON Schema 定义输入，func 签名用 any。

LangChain 的做法：Zod schema 驱动一切——InteropZodType 同时兼容 Zod v3 和 v4，通过 InferInteropZodInput 自动推导 TypeScript 类型。

差异分析：用 Zod 的好处是工具定义可直接序列化为 JSON Schema（用于 function calling 的 API 请求），同时保持类型安全——修改 schema 后 func 签名自动更新，编译时报错而非运行时崩溃。
