🧬 Core 核心抽象

@langchain/core — Runnable / Messages / Language Models / Callbacks / Prompts。整个 LangChain 生态的基石。

核心包v1.1.48~30 模块

1. Runnable — 一切皆可编排

Runnable 是 LangChain.js 最核心的抽象。它是一个泛型基类（base.ts 共 3542 行），定义了三个核心方法：

方法签名用途
invoke(input) → Promise<output>单次调用
stream(input) → AsyncIterable流式输出
batch(inputs[]) → Promise<outputs[]>批量调用

编排原语（所有方法返回新的 Runnable，支持链式调用）：

  • .pipe(next) — 管道组合：a.pipe(b).pipe(c)
  • .bind(config) — 绑定配置参数
  • .withRetry(n) — 自动重试
  • .withFallbacks([...]) — 降级策略
  • .map() — 对输入的每个元素并行执行
  • .withListeners({...}) — 生命周期回调

// ★ Runnable 基类——所有组件的公共祖先
export abstract class Runnable<RunInput, RunOutput, CallOptions>
  extends Serializable
  implements RunnableInterface
{
  protected lc_runnable = true   // ★ 身份标记：用于运行时类型检查
  name?: string

  abstract invoke(
    input: RunInput,
    options?: Partial<CallOptions>
  ): Promise<RunOutput>
}
🧠 Runnable 中的懒执行设计

a.pipe(b) 并不立即执行 b——它只是返回一个新的 Runnable。这个 RunnableSequence 在 invoke 被调用时才按序执行每个步骤。这种惰性求值（lazy evaluation）让 runnable 图可以被先构建再执行，支持图可视化和中间插入。

4. Callbacks 回调系统

回调系统是 LangSmith 可观测性的基础。它通过 CallbackManager 分发事件到多个 handler：

  • handleLLMStart / handleLLMEnd — 模型调用生命周期
  • handleChainStart / handleChainEnd — Chain 执行生命周期
  • handleToolStart / handleToolEnd — 工具调用生命周期
  • 通过 AsyncLocalStorage 在异步上下文中传递回调

5. "如果我来写" 对比思考

🧠 问题：如何保证类型安全的同时保持灵活性？

直觉方案：为每个 Runnable 组合写死类型参数。

LangChain 的做法：使用泛型 + 条件类型。例如 Runnable.bind() 返回的 Runnable 类型会根据绑定参数动态计算输入输出类型。

差异分析：TypeScript 的条件类型让 Runnable 的类型在编译时完全确定——一条 prompt.pipe(model).pipe(parser) 链的类型是自动推导的，无需手动标注。

🧠 问题：如何支持跨运行时的异步上下文传递？

直觉方案：全局变量或 Thread-local storage。

LangChain 的做法：使用 AsyncLocalStorage（Node.js） + 浏览器环境的 polyfill，通过 AsyncLocalStorageProviderSingleton 统一接口。

差异分析：这是 LangChain 支持 Cloudflare Workers / Deno / Bun 等非 Node 运行时的关键基础设施。
