解构 Jotai — 原子化状态管理

Primitive and flexible state management for React · v2.20.0

TypeScript MIT 作者: Daishi Kato

github.com/pmndrs/jotai

1. 项目背景

解决了什么问题

React 内置的 useState/useContext 在小规模足够用，但两个痛点无法回避：

  • Context 的渲染范围问题：单个 Context 值变化时，所有消费者重渲染，无法"按需订阅某个字段"
  • 派生状态的缓存问题：useMemo 手动派生，依赖数组易遗漏

Jotai 的解法：把状态拆成原子（atom）——每个 atom 是最小共享单元，依赖自动追踪，只有直接消费该 atom 的组件会重渲染。

🧠 如果让你来实现...

问题：如何实现"按需订阅原子状态"？

直觉方案：Context + useMemo 精细化控制。缺点：Context 绑定到组件树层级。

Jotai 的做法：绕过 Context。每个 atom 在 Store 中独立注册，store.sub(atom, cb) 订阅单个 atom，组件只在自己关心的 atom 变化时重渲染。

差异分析：把"订阅"从 React 渲染树剥离到 Store 层，消除了层级依赖。

核心特性

  • 零样板：const countAtom = atom(0)，无需 Provider
  • 派生 Atom：atom(get => get(countAtom) * 2)，自动追踪
  • 异步：reader 可返回 Promise，配合 React Suspense
  • 可写派生：atom(read, write) 双向
  • 按需渲染：只订阅用到的 atom

数据流

atom(0) → Store 注册 → useAtomValue(atom) → store.get(atom) + store.sub(atom, cb)
setCount(5) → store.set(atom, 5) → 通知订阅 → useReducer rerender → 新值返回

核心代码：atom() 工厂

src/vanilla/atom.ts:70-90 
let keyCount = 0  // 全局计数器

export function atom(read, write) {
  const key = `atom${++keyCount}`
  const config = { toString() { return key } }

  if (typeof read === 'function') {
    config.read = read  // → 派生 atom
  } else {
    config.init = read   // → 原始值
    config.read = defaultRead
    config.write = defaultWrite
  }
  if (write) config.write = write
  return config
}
💡 设计意图：typeof read === 'function' 一个 if 区分"原始"和"派生"，不需要类继承。全局 keyCount 保证唯一 ID。

核心代码：useAtomValue 订阅机制

src/react/useAtomValue.ts:96-133 
export function useAtomValue(atom, options) {
  const store = useStore(options)

  // useReducer 做带浅比较的强制刷新
  const [[value, ...], rerender] = useReducer(
    (prev) => {
      const next = store.get(atom)
      if (Object.is(prev[0], next)) return prev  // 值没变不渲染！
      return [next, store, atom]
    },
    undefined,
    () => [store.get(atom), store, atom],
  )

  useEffect(() => {
    const unsub = store.sub(atom, () => rerender())  // 只订阅这个 atom！
    return unsub
  }, [store, atom])

  return value
}
💡 设计意图：
1) useReducer 而非 useState：一次调度中同时读值和浅比较，省掉一个 useEffect
2) store.sub(atom, cb) 只订阅这一个 atom——"细粒度按需渲染"的核心
3) Object.is() 避免相同值的无效渲染

Store 内部架构：Building Blocks 系统

internals.ts（38KB）是 Jotai 最复杂的文件。核心设计是 Building Blocks 数组——26 个槽位的函数/状态容器组成的数组。

src/vanilla/internals.ts — buildStore() 核心
type INTERNAL_Store = {
  get<Value>(atom: Atom<Value>): Value
  set<Value, Args>(atom: WritableAtom<Value, Args, Result>, ...args: Args): Result
  sub(atom: AnyAtom, listener: () => void): () => void
}

function buildStore(...partialBuildingBlocks) {
  const store = { get, set, sub }
  const buildingBlocks = [
    new WeakMap(),   // [0] atomStateMap
    new WeakMap(),   // [1] mountedMap
    new WeakMap(),   // [2] invalidatedAtoms
    new Set(),       // [3] changedAtoms
    new Set(),       // [4] mountCallbacks
    new Set(),       // [5] unmountCallbacks
    {},              // [6] storeHooks
    BUILDING_BLOCK_atomRead,            // [7]
    BUILDING_BLOCK_atomWrite,           // [8]
    BUILDING_BLOCK_atomOnInit,          // [9]
    BUILDING_BLOCK_atomOnMount,         // [10]
    ...
    BUILDING_BLOCK_storeGet,            // [21]
    BUILDING_BLOCK_storeSet,            // [22]
    BUILDING_BLOCK_storeSub,            // [23]
    ...
    [0],             // [26] store epoch
  ]
  buildingBlockMap.set(store, Object.freeze(buildingBlocks))
  return store
}
💡 设计意图：
1) 26 个 building blocks 按固定索引访问，不用遍历对象属性——微秒级性能优势
2) partialBuildingBlocks 参数让生态库（jotai-*）覆盖特定 building block 实现，不用继承整个 Store
3) WeakMap 存 atomStateMap 和 mountedMap——atom 对象被 GC 时自动清理状态

Atom 状态追踪与无效化

每个 atom 在 Store 中的状态：

type AtomState<Value> = {
  d: Map<AnyAtom, EpochNumber>  // 依赖图：atom → 对方的 epoch 号
  pendingPromises: Set<AnyAtom>  // 依赖该 atom 且持有 pending Promise 的 atom
  v: Value                       // 当前值
}

派生 atom 执行 read 时，get() 会记录所有被访问的 atom 到 d 中——这就是自动依赖追踪的底层机制。当依赖的 atom 值变化时，它的 epoch 递增，Store 发现不一致后无效化并重新计算派生 atom。

单例 Store 与多实例检测

src/vanilla/store.ts — getDefaultStore()
let defaultStore: Store | undefined
export function getDefaultStore(): Store {
  if (!defaultStore) {
    defaultStore = createStore()
    if (import.meta.env?.MODE !== 'production') {
      (globalThis as any).__JOTAI_DEFAULT_STORE__ ||= defaultStore
      if ((globalThis as any).__JOTAI_DEFAULT_STORE__ !== defaultStore) {
        console.warn('Detected multiple Jotai instances.')
      }
    }
  }
  return defaultStore
}
💡 设计意图：globalThis.__JOTAI_DEFAULT_STORE__ 在非生产环境注入全局变量，检测是否有多个 Jotai 实例同时存在（常见于 monorepo / duplicate dependency），并发出警告。createStore() 还支持 INTERNAL_overrideCreateStore 覆盖，用于测试。

7. 学习要点

可复用的工程实践

  1. typeof read 替代类继承 — 少创建对象
  2. useReducer 带比较的状态刷新 — 省一个渲染周期
  3. Object.is() 防无效渲染 — 成本极低收益巨大
  4. keyCount++ 做唯一 ID — 无需 UUID 库

可借鉴的架构模式

  1. vanilla / React 分层：核心不依赖框架
  2. atom 数据定义，Store 状态容器：职责清晰
  3. infer 泛型推导：ExtractAtomValue 系列确保类型安全
