🧭 PDF.js View UI — 渲染架构与性能优化

深入 PDF.js 的 web/viewer 层，解读大文档渲染策略、LRU 页面缓存、渲染优先级队列及内存管理机制。

web/ mozilla/pdf.js v4.x

一、View UI 概览

PDF.js 的 View UI 层位于 web/ 目录，负责将 Worker 线程生成的渲染数据在浏览器中呈现为可交互的 PDF 阅读器。它不只是一个简单的 canvas 渲染器——更像一个完整的"虚拟滚动 + 按需渲染"引擎。

核心组件：

二、大文档分级策略

PDF.js 根据文档页数自动切换渲染策略，避免打开超大 PDF 时浏览器卡死。

// web/pdf_viewer.js — 常量定义
const PagesCountLimit = {
  FORCE_SCROLL_MODE_PAGE: 10000,  // >1 万页 → 强制滚动模式
  FORCE_LAZY_PAGE_INIT:   5000,   // >5000 页 → 懒初始化
  PAUSE_EAGER_PAGE_INIT:  250,    // >250 页  → 暂停急切初始化
};
页数范围策略行为
≤ 250急切初始化预创建所有页面视图的 DOM 容器，接近原生阅读体验
250–5000暂停急切初始化暂停预创建，改为滚动时按需创建；避免大量 DOM 节点消耗内存
5000–10000懒初始化强制 _scrollMode = PAGE（一次只渲染一页），不预创建任何 DOM
> 10000锁定滚动模式锁定为 PAGE 模式，禁止切换到连续滚动；即使 10000+ 页也能流畅翻页
🧠 如果让你来实现…

直觉方案：一次性创建所有页面视图的 DOM，然后按需渲染 Canvas。

PDF.js 的做法：不仅 Canvas 按需渲染，连 DOM 也按需创建。超过 250 页就停止预创建，超过 5000 页连滚动容器都切换为单页模式。

差异分析：直觉方案在 1000 页以上会因大量 DOM 节点（每页至少 3-4 个 div）导致浏览器内存飙升、Layout 性能崩溃。PDF.js 的三级策略是一个经典的"graceful degradation"设计。

三、渲染优先级队列

PDFRenderingQueue 是整个渲染系统的大脑。它决定"当前该渲染哪一页"，并在空闲时触发资源清理。

3.1 优先级排序

// web/pdf_rendering_queue.js — getHighestPriority()
/**
 * Priority:
 * 1. visible pages             ← 用户视野内的页面
 * 2. zoomed-in partial views   ← 局部放大视图
 * 3. next page (scroll dir)    ← 滚动方向的下一页
 * 4. preRenderExtra            ← 再下一页（spread 模式下）
 * 5. "hole" pages              ← 可见范围中的空位
 */
💡 "hole" 页面

在 spread 模式（双页视图）的高缩放级别下，可见页面 ID 可能不连续（例如显示第 3 页和第 5 页，第 4 页是"hole"）。队列会优先填补这些空位。

3.2 快速滚动保护

当用户快速滚动或缩放时，PDF.js 会智能跳过某些渲染以保持响应性：

const ignoreDetailViews =
  this.#scaleTimeoutId !== null ||   // 正在缩放 → 跳过局部视图
  (this.#scrollTimeoutId !== null && // 快速滚动 → 跳过被取消的局部视图
    visiblePages.views.some(p => p.detailView?.renderingCancelled));

用户缩放/快速滚动时，局部缩放视图（detailView）的重渲染会被跳过。等用户停止操作后，定时器清零，再完整渲染。

3.3 空闲清理

// web/pdf_rendering_queue.js
const CLEANUP_TIMEOUT = 30000; // 30 秒

renderHighestPriority(currentlyVisiblePages) {
  // ...渲染逻辑...
  if (this.onIdle) {
    this.#idleTimeout = setTimeout(
      this.onIdle.bind(this),
      CLEANUP_TIMEOUT
    );
  }
}

30 秒内没有任何渲染任务时，触发 onIdle 回调执行页面清理（pdfPage.cleanup()），释放 Worker 线程中缓存的页面数据。

四、LRU 页面视图缓存

PDF.js 使用 PDFPageViewBuffer 实现 LRU 缓存，控制同时存在的页面视图数量，防止内存无限增长。

// web/pdf_viewer.js
const DEFAULT_CACHE_SIZE = 10; // 默认缓存 10 个页面视图

class PDFPageViewBuffer {
  #buf = new Set();  // Set 保持插入顺序，天然支持 LRU
  #size = 0;

  push(view) {
    const buf = this.#buf;
    if (buf.has(view)) {
      buf.delete(view); // 移到末尾 = 标记为最近使用
    }
    buf.add(view);
    if (buf.size > this.#size) {
      this.#destroyFirstView(); // 淘汰最旧的
    }
  }

  resize(newSize, idsToKeep = null) {
    this.#size = newSize;
    if (idsToKeep) {
      // 将指定页面移到末尾，推迟销毁
    }
    while (this.#buf.size > this.#size) {
      this.#destroyFirstView();
    }
  }
}

设计亮点：利用 ES6 Set 的插入顺序特性实现 LRU，无需额外维护链表。当 view 已在缓存中时，delete + add 将其移到末尾（标记为最近使用）。

🧠 如果让你来实现…

直觉方案：用数组 + splice 实现 LRU，或引入第三方 LRU 库。

PDF.js 的做法：直接用 new Set()，利用 Set 的顺序保证。极简代码，零依赖。

差异分析：Set 的 delete + add 是 O(1)，而数组 splice 是 O(n)。这是一个被低估的 JavaScript 技巧：Set 是天然的 LRU 数据结构。

缓存在 visible 事件中更新：页面变为可见时 push 到 buffer，不可见且不在 buffer 中的页面调用 pdfPage.cleanup() 释放 worker 数据。

五、内存优化方案

PDF.js 的内存优化贯穿多个层级，从页面视图到 Worker 线程都有严格的清理机制。

5.1 页面视图生命周期

// web/pdf_page_view.js
destroy() {
  this.reset();           // 取消渲染、清空 layer
  this.pdfPage?.cleanup(); // 通知 Worker 释放该页数据
}

deleteMe(isCut) {
  if (isCut) {
    this.div.remove();   // 裁剪模式下直接移除 DOM
    return;
  }
  // 正常模式：重置视图状态但保留 DOM 结构
  this.reset();
}

关键点：destroy() ≠ deleteMe()。前者在页面被 LRU 淘汰时调用（释放 worker 数据），后者在页面离开视口时调用（保留 DOM 容器以便下次快速恢复）。

5.2 Canvas 像素限制

参数默认值作用
maxCanvasPixels4096 × 8192 = 32M单页 Canvas 最大像素数，防止超大页面耗尽 GPU 内存
maxCanvasDim32767Canvas 单维度最大像素（宽或高），浏览器硬限制
⚠️ 易错点

在高 DPI 屏幕上渲染大页面时，outputScale 可能使 Canvas 像素数乘以 4（2x 设备像素比），可能导致超过 maxCanvasPixels 上限而触发降级渲染。

5.3 Worker 线程数据清理

// web/pdf_viewer.js — 缩略图渲染完成后触发清理
this.eventBus.on("thumbnailrendered", ({ pageNumber, pdfPage }) => {
  const pageView = this._pages[pageNumber - 1];
  if (!this.#buffer.has(pageView)) {
    pdfPage?.cleanup(); // 不在缓存的页面 → 释放 worker 数据
  }
});

缩略图渲染是一个"钩子"：渲染完缩略图后，如果对应页面视图不在 LRU 缓存中，立即清理 worker 数据，避免"缩略图加载了但页面视图已销毁"导致的内存泄漏。

六、Canvas 绘制优化

6.1 OutputScale 自适应

// web/pdf_page_view.js
#computeScale() {
  const outputScale = (this.outputScale = new OutputScale());
  // outputScale.sx / sy 根据 devicePixelRatio 自动计算
}

在高 DPI 屏幕上，outputScale 自动提升渲染分辨率，确保文字清晰；在低端设备上可限制缩放避免性能问题。

6.2 位图缓存

// src/display/canvas.js — CanvasGraphics
this._cachedBitmapsMap = new Map(); // 缓存已解码的图片/位图
this._cachedScaleForStroking = [-1, 0];
this._cachedGetSinglePixelWidth = null;

PDF 中的图片对象（JPEG/PNG/位图）解码成本高，CanvasGraphics 用 Map 缓存已解码数据，同一图片在多个页面上复用时直接命中缓存。

6.3 pauseOnVisibility

当浏览器标签页切换到后台时，rendering queue 暂停渲染，减少 CPU 占用。

七、关键优化点总结

🧠 架构启示

PDF.js View UI 的设计可以看作一个迷你版的"虚拟列表 + 按优先级渲染引擎"。它的核心思想——先渲染可见的、再预渲染可能需要的、最后清理不需要的——不仅适用于 PDF 阅读器，任何需要渲染大量内容的 Web 应用都可以借鉴。
