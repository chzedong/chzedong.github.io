核心设计原则：Worker 负责 CPU 密集的解析和求值，主线程只做轻量的 Canvas 绘制和 DOM 操作。两者通过 postMessage 传输 OperatorList（使用 Transferable Objects 零拷贝）。

二、Worker 架构

lib/core/worker.js:47-220

Worker 端有两层抽象：

  • WorkerTask — 单个异步任务的 Promise 封装，支持 terminate() 安全取消
  • WorkerMessageHandler — Worker 的入口，注册消息处理器
lib/core/worker.js:76-93 
class WorkerMessageHandler {
  static setup(handler, port) {
    handler.on("test", data => {          // ★ 传输协议测试
      handler.send("test", data instanceof Uint8Array);
    });
    handler.on("configure", data => {     // ★ 日志级别配置
      (0, _util.setVerbosityLevel)(data.verbosity);
    });
    handler.on("GetDocRequest", data => { // ★ 核心：文档加载请求
      return WorkerMessageHandler.createDocumentHandler(data, port);
    });
  }
}

createDocumentHandler() 在 Worker 中创建完整的 PDF 解析管线：NetworkManager → LocalPdfManager → XRef → Catalog → PartialEvaluator。它向主线程暴露以下 RPC 方法：

  • GetOperatorList — 获取页面的图形指令序列
  • GetTextContent — 获取页面的文本内容（用于搜索和选择）
  • GetAnnotations — 获取页面批注
  • RenderPage — 直接请求 Worker 渲染（Node.js 模式）
  • GetData / SaveDocument — PDF 数据读写
🧠 如果让你来实现 Worker 通信

问题：主线程和 Worker 之间需要双向 RPC（请求-响应）。原生 postMessage 是单向消息传递，如何建立请求-响应的关联？

直觉方案：每条消息带一个自增 ID，Worker 响应时带回相同 ID，主线程维护一个 pending 的 Promise Map。

该库的实际做法：正是如此。但 pdf.js 额外实现了流式传输——对于 OperatorList 等大量数据，使用 sendStreamRequest 将数据分块传输，支持渐进而非一次性发送。这样主线程可以边接收边渲染，大幅降低首屏延迟。

差异分析：直觉方案适用于请求-响应的 RPC，但不适合流式场景。pdf.js 的 MessageHandler 在基础 RPC 上增加了 StreamSink 抽象——Worker 可以分多次 send 同一个流的数据和完成信号，主线程通过 ChunkedStream 逐步读取。

三、MessageHandler — 双向 RPC 引擎

lib/shared/message_handler.js

MessageHandler 封装了 postMessage，提供了：

  • send(name, data, transfers) — 发送消息（支持 Transferable）
  • sendWithPromise(name, data, transfers) — 发送并返回 Promise
  • sendWithStream(name, data, transfers) — 发送并返回可读流
  • on(name, callback) — 注册消息处理器

每个请求包含一个唯一的 streamId。当请求完成时，Worker 发送 streamId: null 表示流结束。主线程的 MessageHandler 通过 PromiseCapability（自实现的 Deferred）管理请求生命周期。

💡 隐蔽但关键：Array.prototype 污染检测

在 worker.js:110-116，Worker 启动时会检查 Array.prototype 上是否存在可枚举属性。因为 for...in 遍历数组会被 polyfill 污染（例如 core-js），而 pdf.js 内部大量使用 TypedArray，数组原型被污染会导致灾难性的渲染错误。如果检测到污染，直接抛出错误。

四、API 层详解

lib/display/api.js:83-200 — PDF.js 对外的门面

入口函数 getDocument(src)：

function getDocument(src) {
  const task = new PDFDocumentLoadingTask();
  let source;

  if (typeof src === "string" || src instanceof URL) {
    source = { url: src };                // ★ 支持 URL 字符串
  } else if ((0, _util.isArrayBuffer)(src)) {
    source = { data: src };               // ★ 支持 ArrayBuffer
  } else if (src instanceof PDFDataRangeTransport) {
    source = { range: src };              // ★ 支持 Range 请求（大文件流式加载）
  } else {
    source = src;                         // ★ 支持对象字面量（含 password, cMapUrl 等）
  }

  const worker = new PDFWorker();           // ★ 自动创建 Worker
  const params = { source, worker, ... };
  const messageHandler = new MessageHandler(...);
  messageHandler.send("GetDocRequest", params);
  return task;                           // ★ 立即返回 loadingTask，DocumentProxy 在 .promise 中
}

核心代理对象：

对象职责关键方法
PDFDocumentLoadingTask加载任务句柄.promise → 等待加载完成；.destroy() → 取消加载

🏗️ 「Promise 代理」模式

PDF.js 的 API 设计是经典的 Proxy 模式：用户代码永远不直接访问 Worker 内部对象。所有交互通过 xxxProxy 进行，这些 Proxy 内部封装了 MessageHandler 的 RPC 调用。这使得 API 层可以在不改变用户代码的情况下切换底层实现（例如从 Worker 模式切换到 LoopbackPort 同线程模式）。

五、CanvasGraphics — Canvas 渲染器

lib/display/canvas.js:27 — 2810 行

CanvasGraphics 是 PDF.js 渲染管线的终端——它接收 OperatorList，按顺序执行每个操作码，最终在 Canvas 2D Context 上绘制出 PDF 内容。

核心方法 executeOperatorList()：

// canvas.js 的执行循环（简化）
const EXECUTION_TIME = 15;     // ★ 每帧最多执行 15ms
const EXECUTION_STEPS = 10;   // ★ 每批最多 10 个操作符

executeOperatorList(operatorList, ...) {
  const fnArray = operatorList.fnArray;   // [OPS.xxx, OPS.yyy, ...]
  const argsArray = operatorList.argsArray; // [[args], [args], ...]

  let i = 0;
  const next = () => {
    const startTime = Date.now();
    let steps = 0;
    while (i < fnArray.length) {
      this.executeOp(fnArray[i], argsArray[i]); // ★ 执行单个操作
      i++;
      steps++;
      if (steps >= EXECUTION_STEPS && Date.now() - startTime > EXECUTION_TIME) {
        break;  // ★ 时间片用完，让出主线程
      }
    }
    if (i == fnArray.length) {
      resolve(); // 完成
    } else {
      setTimeout(next, 0);  // ★ 下一帧继续
    }
  };
  next();
}

这个时间片分块执行策略是 PDF.js 渲染不卡顿的核心原因。对于含数万个图形指令的复杂页面，它会自动拆分为多个微任务，确保每帧不超过 15ms 的渲染预算。

⚠️ 容易忽视的渲染细节

JPEG 直通（JPEG Pass-through）：当 PDF 内嵌 JPEG 图像时，CanvasGraphics 不会先解码再重新编码——它直接将 JPEG 二进制数据传给 ctx.drawImage()（通过 createImageBitmap 或 Blob URL）。这避免了 JPEG 二次压缩的质量损失，也是 PDF.js 渲染质量媲美原生阅读器的原因之一。

Canvas 镜像（Mirroring）：可选内容组（图层）的显示/隐藏通过「Canvas 镜像」实现——创建一个不可见的 Canvas 作为主 Canvas 的同步副本。所有绘制操作同时写入两个 Canvas，当用户切换图层时，只需显示/隐藏对应的 DOM Canvas 元素，无需重绘。

六、TextLayer 和 AnnotationLayer

PDF 渲染不只是像素绘制。为了支持文本选择、搜索和可访问性，PDF.js 在 Canvas 之上叠加了两个 DOM 层：

图层文件职责
TextLayerlib/display/text_layer.js
lib/web/text_layer_builder.js		在 Canvas 上方创建透明的 <span> 元素，每个 <span> 精确定位到 Canvas 中对应字符的位置，允许用户选择和复制文本
AnnotationLayerlib/display/annotation_layer.js
lib/web/annotation_layer_builder.js		渲染链接、表单控件、高亮批注等交互元素。使用 HTML <a>、<input> 等原生元素，完全可访问
XfaLayerlib/display/xfa_layer.jsXFA（XML Forms Architecture）表单渲染，将 Adobe LiveCycle 表单转换为 HTML

TextLayer 的核心挑战是坐标对齐：Canvas 中的文本位置使用 PDF 坐标系（原点在左下角，单位为 points），而 DOM 使用视口坐标系（原点在左上角，单位为 CSS pixels）。TextLayer 需要根据页面的缩放比例和变换矩阵精确计算每个 span 的 CSS left/top/font-size，使选择高亮与 Canvas 中的文本精确对齐。

🧠 如果你来实现文本选择

问题：Canvas 中的文本是像素，无法被浏览器选择。如何让用户选择和复制 PDF 中的文字？

直觉方案：自己实现选择逻辑（鼠标拖拽 → 计算选中区域 → 提取对应文本）。

该库的实际做法：在 Canvas 上方叠加透明的 HTML 元素，利用浏览器原生选择行为。这是一种巧妙的「用平台能力代替自己实现」的思路。

差异分析：自己实现选择逻辑需要处理 RTL 文本、连字、换行等无数边界情况。叠加透明 DOM 虽然增加了坐标对齐的复杂性，但获得了免费的键盘导航、屏幕阅读器支持和国际化。

七、SVG 渲染

lib/display/svg.js — SVGGraphics

SVGGraphics 与 CanvasGraphics 共享相同的 OperatorList 输入，但输出是 SVG DOM 元素而非 Canvas 像素。这展示了 OperatorList 作为平台无关中间表示的价值。

SVG 渲染的每个操作符对应一个 SVG 元素：showText → <text>、paintImageXObject → <image>、路径操作 → <path>。

主要应用场景：Node.js 无 Canvas 环境的 PDF 渲染、需要矢量输出的打印场景。

💡 Viewer 的松耦合设计

Viewer 的各组件通过 EventBus 通信，互相不直接引用。这意味着你可以轻松替换任何一个组件——例如用自定义的搜索 UI 替换 PDFFindBar，只需监听相同的事件即可。这种基于事件的架构在 MVC 框架流行之前是前端架构的经典范式。
