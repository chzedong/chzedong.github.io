一、数据流总览

PDF 解析是从原始字节到结构化数据的过程，可以类比为编译原理中的「词法分析 → 语法分析 → 语义分析 → 中间代码生成」。PDF.js 的核心解析管线如下：

二、PDF 文件结构速览

理解 PDF.js 的解析逻辑，首先要理解 PDF 文件本身的四段结构：

# 1. 文件头
%PDF-1.4

# 2. 对象集合（Body）
1 0 obj
<< /Type /Catalog /Pages 2 0 R >>
endobj

2 0 obj
<< /Type /Pages /Kids [3 0 R] /Count 1 >>
endobj

3 0 obj
<< /Type /Page /Parent 2 0 R /Contents 4 0 R /MediaBox [0 0 612 792] >>
endobj

4 0 obj
<< /Length 44 >>
stream
BT /F1 12 Tf 100 700 Td (Hello PDF) Tj ET
endstream
endobj

# 3. 交叉引用表（XRef）
xref
0 5
0000000000 65535 f 
0000000015 00000 n 
...

# 4. 文件尾（Trailer）
trailer
<< /Size 5 /Root 1 0 R >>
startxref
492
%%EOF
📌 PDF 的核心抽象

PDF 文件本质是一个对象数据库：通过交叉引用表（XRef）定位每个对象在文件中的字节偏移，通过 Trailer 找到根对象（Catalog），再通过字典引用（/Pages 2 0 R）遍历整个对象图。

三、Lexer — 词法分析器

lib/core/parser.js:73

Lexer 是解析的第一阶段，将原始字节序列化为 Name、Num、String、Cmd 等 PDF 基本 Token。它不关心语义——只回答「下一个 Token 是什么」。

核心设计：两字符前瞻（two-token lookahead）。Parser 通过 buf1 和 buf2 维护当前 Token 和下一个 Token 的缓冲区，这允许它在遇到 <<（字典开始）时立即切换到字典解析模式。

lib/core/parser.js:73-101 
class Parser {
  constructor({ lexer, xref, allowStreams = false, recoveryMode = false }) {
    this.lexer = lexer;
    this.xref = xref;
    this.allowStreams = allowStreams;
    this.recoveryMode = recoveryMode;
    this.refill();  // ★ 预取前两个 Token
  }

  refill() {
    this.buf1 = this.lexer.getObj();
    this.buf2 = this.lexer.getObj();
  }

  shift() {
    if (this.buf2 instanceof Cmd && this.buf2.cmd === "ID") {
      this.buf1 = this.buf2;  // ★ 内联图像特殊处理
      this.buf2 = null;
    } else {
      this.buf1 = this.buf2;
      this.buf2 = this.lexer.getObj();
    }
  }
}
🧠 如果让你来实现 Lexer

问题：PDF 的 Token 类型边界由空格、括号、尖括号等分隔符决定。如何高效地从字节流中提取 Token？

直觉方案：逐字节扫描，状态机切换模式（normal/string/hex/name）。

该库的实际做法：确实使用状态机，但额外实现了「inline image 的 back-channel」——遇到 BI（Begin Image）命令时，Lexer 不尝试解析图像数据（那可能是二进制乱码），而是原样返回后续字节，由 Parser 调用专门的 makeInlineImage() 处理。这是一种「解析器指导词法分析器」的反向控制流。

差异分析：如果 Lexer 独立处理内联图像，需要知道图像数据何时结束（即遇到 EI 标记）。但在图像二进制数据中 EI 字节序列可能自然出现，导致误匹配。PDF.js 的做法是把图像边界的判断权交给更上层的 Parser，避免了 Lexer 层面的假阳性。

四、Parser — 递归下降解析器

lib/core/parser.js:116-320

Parser 的核心方法是 getObj()，它是一个递归下降解析器：根据当前 Token 类型调度到不同的解析分支。PDF 的数据结构（数组、字典、流、间接引用）天然适合递归解析。

// parser.js: getObj() 的核心调度逻辑（简化）
getObj(cipherTransform = null) {
  const buf1 = this.buf1;
  this.shift();

  if (buf1 instanceof Cmd) {
    switch (buf1.cmd) {
      case "BI":         return this.makeInlineImage(...);   // 内联图像
      case "[":           return this.parseArray(...);         // 数组
      case "<<":          return this.parseDict(...);          // 字典
      case "stream":     return this.parseStream(...);        // 流
      case "endobj":     return null;                       // 对象结束
    }
  }

  if (buf1 instanceof Num) {
    const num = buf1;
    const gen = this.buf1;
    if (gen instanceof Num && this.buf2 instanceof Cmd
        && this.buf2.cmd === "R") {
      // ★ "1 0 R" = 间接引用（ref）
      this.shift(); this.shift();
      return Ref.get(num, gen);
    }
    return num;
  }

  if (buf1 instanceof Name) return buf1;
  if (typeof buf1 === "string") return buf1;
}

这个递归下降结构让 Parser 能自然处理嵌套结构：[ 1 0 R << /A [ 2 0 R ] >> ]——解析时会递归进入数组和字典。

💡 设计亮点：错误恢复（Recovery Mode）

PDF 文件常常不完美（由各种 PDF 生成器产生）。Parser 支持 recoveryMode：当遇到 EOF 时，不立即抛出异常，而是尝试返回已解析的部分结果。这在处理损坏的 PDF 时至关重要。

五、XRef — 交叉引用表

lib/core/xref.js:41-200

XRef 是 PDF 的「对象目录」。每个 PDF 对象都有唯一的对象编号（object number）和代数（generation），XRef 存储了每个对象在文件中的字节偏移。

核心结构：

  • entries[] — 对象入口数组，每个入口包含 { offset, gen, uncompressed }
  • _cacheMap — 已解析对象的 Map 缓存，避免重复解析
  • _pendingRefs — 正在解析中的引用集合（防止循环引用）

parse() 方法的工作流程：

  1. 从文件尾 startxref 找到交叉引用表的偏移量
  2. readXRef() 解析传统 XRef 表或 XRef 流
  3. 回退到 indexObjects()（在全文件扫描所有 obj ... endobj）——仅在 recoveryMode 下
  4. 提取 Trailer 字典中的 /Root（Catalog 引用）和 /Encrypt（加密信息）
lib/core/xref.js:69-98 
parse(recoveryMode = false) {
  let trailerDict;
  if (!recoveryMode) {
    trailerDict = this.readXRef();      // 正常路径：读 XRef 表
  } else {
    (0, _util.warn)("Indexing all PDF objects");
    trailerDict = this.indexObjects();  // ★ 恢复路径：全文件扫描
  }

  trailerDict.assignXref(this);
  this.trailer = trailerDict;

  let encrypt;
  try {
    encrypt = trailerDict.get("Encrypt");   // ★ 检查 PDF 是否加密
  } catch (ex) { ... }

  if (encrypt instanceof Dict) {
    // 提取加密参数，创建 CipherTransformFactory
    this.encrypt = new CipherTransformFactory(encrypt, fileId, this.pdfManager.password);
  }

  let root = trailerDict.get("Root");       // 获取 Catalog 引用
  if (root instanceof Dict) {
    const pages = root.get("Pages");
    if (pages instanceof Dict) return;
  }
}

🏗️ 架构洞察：惰性加载（Lazy Loading）

XRef 只存储偏移量，不立即解析对象内容。只有当代码通过 xref.fetch(ref) 访问某个对象时，才会从文件流中读取并解析。这是 PDF.js 能处理超大 PDF 的关键——它可以只加载当前页面需要的对象（字体、图像、内容流），而不需要将整个文件读入内存。

六、Catalog — 文档目录树

lib/core/catalog.js:59-180

Catalog 是 PDF 文档的「根节点」。它从 Trailer 的 /Root 引用开始，构建出完整的页面树。Catalog 不仅提供页面访问，还管理：

  • 页面树遍历 — getPage() / getPageIndex()
  • 缓存管理 — fontCache、builtInCMapCache、standardFontDataCache
  • 文档元数据 — 版本号、语言、表单（AcroForm）、大纲、附件
  • 可选内容组 — 图层的显示/隐藏
🧠 如果让你来实现页面树遍历

问题：PDF 的页面结构是一个树（Pages 节点可以有子 Pages 和 Page节点），如何根据页号快速定位？

直觉方案：遍历树建立 page-index → page-ref 的映射表。

该库的实际做法：使用 pageKidsCountCache（以 RefSetCache 实现）缓存每个 Pages 节点的子页面数。通过二分查找在树中定位目标页，避免全树遍历。同时 pageIndexCache 缓存已计算的页面索引。

差异分析：直觉方案在 10000 页的 PDF 中需要遍历所有节点，O(n) 复杂度。实际方案利用子页面计数的缓存实现 O(log n) 的树搜索，大幅减少了大型文档的定位开销。

七、PartialEvaluator — 内容流求值引擎

lib/core/evaluator.js:27 — 4610 行，PDF.js 中最复杂的模块

PartialEvaluator 是解析链路的最后一环，它的输入是页面内容流（经过解码的 PDF 图形操作序列），输出是 OperatorList（平台无关的图形指令序列）。

工作流程：

  1. getOperatorList() 接收页面内容流
  2. 内部再次调用 Parser 解析内容流中的 PDF 图形操作符
  3. EvaluatorPreprocessor 预处理操作符（处理 q/Q 图形状态保存/恢复）
  4. 对每个操作符分派到对应的 op_xxx 处理函数
  5. 处理函数将其翻译为 OperatorList 条目
// evaluator.js 中的操作符分派表（简化的 OPS 常量）
const OPS = {
  paintFormXObject: 0,
  beginText: 1,
  setFont: 2,
  showText: 3,
  paintImageXObject: 4,
  paintInlineImageXObject: 5,
  setFillColor: 6,
  setStrokeColor: 7,
  setLineWidth: 8,
  // ... 共 100+ 个操作符
};

// 对 PDF 操作符 "Tj"（show text）的处理映射
case "Tj": {   // 显示文本字符串
  const text = getTextContent();    // ★ 从字体编码提取 Unicode
  operatorList.addOp(OPS.showText, [text, fontName, fontSize, matrix]);
  break;
}
⚠️ 性能关键：为何叫 "Partial" Evaluator

PDF 内容流中的 Form XObject 可以引用其他页面资源。PartialEvaluator 在遇到它们时不立即求值，而是将其延迟到渲染阶段（CanvasGraphics 中）。这种部分求值策略避免了过早展开导致的内存爆炸，同时允许渲染器根据视口裁剪决定是否跳过不可见元素。

八、OperatorList — 平台无关的中间表示

lib/core/operator_list.js

OperatorList 是一个简单的线性数据结构：[opCode, arg1, arg2, ..., opCode, arg1, ...]。每个操作码后跟零个或多个参数。它是 Worker 线程和主线程之间的传输格式。

设计特性实现优势
紧凑性使用数字操作码（OPS 枚举）而非字符串减少 postMessage 序列化开销
可传输性使用 Transferable Objects（TypedArray）传递图像数据零内存拷贝，大幅提升大图传输性能
平台无关性不包含任何 Canvas/DOM 特定操作同一份 OpsList 可在 Canvas、SVG、Node.js 中渲染
可分块性支持 addDependency() 标记需要异步加载的资源支持渐进式渲染——先画文本，图像加载完成后再补绘

🏗️ 为什么选择线性列表而非 AST？

PDF 的图形操作本质上是顺序指令（类似 PS/PDF 操作符）。线性 OperatorList 比 AST 更适合这个场景：

  • 渲染器只需顺序遍历，不需要递归遍历树
  • 序列化更简单，适合 postMessage 传输
  • 可以轻松打断/恢复（按时间片分块渲染，避免长任务阻塞主线程）

九、设计模式总结

模式应用位置为何选择
递归下降解析器Parser.getObj()PDF 的嵌套结构（数组/字典/流）天然适合递归下降
享元模式（Flyweight）Name、Cmd、Ref 使用静态缓存工厂PDF 中名称和命令重复率极高，共享实例节省内存（如 /Type 出现数千次）
惰性加载（Lazy）XRef.fetch()按需加载对象，避免大型 PDF 全部加载到内存
代理模式（Proxy）LocalPdfManager / NetworkPdfManager统一本地文件和远程文件的访问接口
策略模式（Strategy）各种 Stream 解码器不同压缩算法（Flate/LZW/JPEG）可互换，通过 Dict.get("Filter") 选择
状态模式EvaluatorPreprocessorPDF 图形状态栈（q/Q 操作符），保存/恢复变换矩阵、颜色、裁剪路径
🧠 最值得学习的设计

整个解析链路的分层架构——Lexer（字节→Token）→ Parser（Token→对象）→ XRef（偏移→对象）→ Evaluator（对象→指令）→ OperatorList（指令→传输）。每一层只做一件事，上层完全不需要知道下层的实现细节。这种分层在 JavaScript 项目中极为罕见，是系统级编程思维的体现。
