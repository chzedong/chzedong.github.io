🔤 字体系统与流编解码

PDF.js 最深度的两个技术领域：字体工程（Type1/CFF/TrueType 解析与渲染）和流处理（Flate/LZW/JPEG/JBIG2 编解码器链）。

fonts.js: 2958行 cff_parser.js parser.js 文件头 ~20 个流解码器

一、字体系统总览

字体是 PDF 渲染中最复杂的子系统。PDF 文件可以内嵌多种格式的字体数据，PDF.js 需要在纯 JavaScript 中解析并渲染这些字体。这相当于在浏览器中实现了一个迷你字体引擎。

三、CFF 字体解析 — PDF.js 中最精密的解析器

lib/core/cff_parser.js — CFF（Compact Font Format）解析器

CFF 是 OpenType/CFF 字体的核心格式，也是 PDF 中最常见的 Type1C 字体格式。它的设计目标是在保持 Type1 的全部表现力的同时，使用更紧凑的二进制编码（Type2 CharString）。

CFF 文件结构：

# CFF 数据的核心段
Header   — 版本、hdrSize、offSize
Name     — 字体名索引
Top DICT — 全局字体参数（CharStrings 偏移、CharSet 偏移、私有字典偏移等）
String   — 字符串索引
Global Subrs — 全局子程序（可被 CharString 调用）
CharStrings  — 每个字形的 Type2 字节码
Private DICT — 私有参数字典
Local Subrs  — 局部子程序

🏗️ CFF 解析的核心挑战：子程序展开（Subroutinization）

Type2 CharString 支持 callsubr 和 callgsubr 操作符——在字形定义中调用局部或全局子程序。这类似于编程语言的函数调用。CFFParser 需要在解析时展开所有子程序调用，同时还要处理 flex 机制（Type2 的平滑曲线优化）和提示指令（hints）——这些提示告诉光栅化器如何在小字号时对齐笔划到像素网格。

PDF.js 的策略是：解析时展开子程序并忽略提示（因为浏览器渲染引擎有自己的提示策略）。这使得 CFF 解析后得到的是扁平的纯贝塞尔曲线路径，可以直接交给 Canvas 绘制。

🧠 如果让你来实现 CFF 解析

问题：CharString 中的操作符宽度编码是变长的——有的操作符是 1 字节、有的是 2 字节。如何高效解析？

直觉方案：维护一个大的 switch-case 映射表。

该库的实际做法：使用 256 长度的查找表，其中 1 字节操作符直接映射，0x0C（escape）后面跟的字节作为 2 字节操作符。解析器逐字节扫描，遇到数字压栈（Type2 对整数有特殊的编码），遇到操作符就执行对应的绘制函数。

差异分析：Type2 的整数编码是精心设计的空间优化——-107 到 +107 范围内的数字编码为 1 字节，超出范围使用 2-5 字节。这比直接存储 32-bit 浮点数节省了 70%+ 的空间。解析这种编码是理解「二进制格式设计」的优秀案例。

四、CMap — 字符映射表

lib/core/cmap.js

CMap（Character Map）将 PDF 内容流中的字符代码（通常 1-2 字节）映射到字形索引或 Unicode 码点。这是 PDF 文本正确显示的关键。

三种映射方式：

方式机制适用场景
内置编码StandardEncoding、MacRomanEncoding、WinAnsiEncoding拉丁字体
CMap 流使用 bfrange/bfchar 操作符定义范围映射CJK 字体（中日韩）
ToUnicode CMap从字符代码直接映射到 Unicode所有字体（用于文本提取和搜索）

CMap 解析器的核心是一个二分查找树：bfrange 定义的连续范围如 <0041> <005A> <0061>（A-Z → a-z），bfchar 定义的点映射如 <0041> <0041>，解析后形成一个排序的区间数组，查询时二分查找。

五、Unicode 映射与文本提取

lib/core/to_unicode_map.js + lib/core/unicode.js

PDF 的文本提取之所以复杂，是因为 PDF 不是为了文本提取而设计的。PDF 内容流中存储的是字形索引（glyph index）而非字符编码。要将屏幕上的像素变成可搜索/可复制的 Unicode 文本，需要三层映射：

  1. 字符代码 → CID/GID（通过 CMap）
  2. CID/GID → Unicode（通过 ToUnicode CMap）
  3. 字形名称 → Unicode（通过 Adobe Glyph List 回退——当 ToUnicode CMap 缺失时）

glyphlist.js 包含 Adobe 标准的 ~4300 个字形名称到 Unicode 的映射表（例如 Aacute → U+00C1）。unicode.js 包含 Unicode 属性查询（如判断字符是否属于 CJK 范围、是否是结合字符、RTL 语言方向判断等）。

⚠️ 常见陷阱：ToUnicode 的不可靠性

许多 PDF 生成器提供的 ToUnicode CMap 是不完整的甚至是错误的。当 ToUnicode 映射失败时，PDF.js 会回退到字形名称（通过 glyphlist.js）和启发式 Unicode 推断。这使得 PDF.js 的文本提取比大多数竞品更准确，即使面对「糟糕的」PDF 文件。

六、FontRenderer — 从字形轮廓到 Canvas 路径

lib/core/font_renderer.js

当 PDF 字体无法作为系统字体使用时（例如内嵌的 Type1/CFF/Type3 字体），FontRenderer 负责将字形轮廓转换为 Canvas 可绘制的路径。它的输入是 CFF/Type1 解析后的扁平字形数据（一系列 moveTo/lineTo/curveTo 指令），输出是 Canvas 2D 的路径操作。

字形光栅化的两种模式：

  1. 路径模式 — 使用 ctx.fill() 直接在 Canvas 上绘制字形轮廓。质量最高但较慢
  2. 字体代理模式（Font Face） — 将解析出的 OpenType 字体通过 FontFace API 注册为浏览器字体，然后使用 ctx.fillText() 绘制。利用浏览器原生文字渲染（含 subpixel antialiasing），质量和性能均更优

v2.16.105 版本默认使用字体代理模式，通过 opentype_file_builder.js 将 PDF 字体数据重新打包为符合 OpenType 规范的字体文件。

七、流编解码总览

PDF 中的几乎所有数据（页面内容、字体、图像）都以流（Stream）的形式存储——一个字典（描述如何解码）+ 一段二进制数据。PDF.js 的流处理采用管道（Pipeline）模式：每个解码器是一个可组合的流转换器。

八、解码器链 — 可组合的流转换

lib/core/parser.js:30-55 — 解码器注册表

PDF 使用 /Filter 和 /DecodeParms 字典字段指定解码方式。多个 filter 可以串联（例如先 ASCII85 解码，再 Flate 解压）。PDF.js 通过策略模式实现可互换的解码器：

解码器文件算法复杂度
FlateDecodeflate_stream.jszlib/deflate（使用浏览器原生 DecompressionStream 或 pako）最常用
LZWDecodelzw_stream.jsLempel-Ziv-Welch（GIF 同款算法）较旧，但经典
ASCII85Decodeascii_85_stream.jsBase85 编码，每 4 字节 → 5 个可打印 ASCII 字符Adobe 专用
ASCIIHexDecodeascii_hex_stream.js十六进制编码简单
RunLengthDecoderun_length_stream.js游程编码（RLE）单色图像常用
Predictorpredictor_stream.jsPNG 预测器（None, Sub, Up, Avg, Paeth）Flate 之后使用
CCITTFaxDecodeccitt_stream.jsGroup 3 / Group 4 传真编码黑白文档扫描

解码器的组合由 Parser.makeStream() 在运行时动态构建：读取字典的 /Filter 字段，按照数组顺序逐层包装，形成 FlateStream(ASCII85Stream(DecryptStream(RawStream))) 的嵌套结构。每层只负责自己的解码逻辑，完全符合装饰器模式。

🧠 如果让你来实现解码器组合

问题：PDF 的 Filter 数组可能是 [/ASCII85Decode /FlateDecode]，也可能是单个 /FlateDecode。如何优雅地组合解码器？

直觉方案：if-else 或 switch 判断，手动嵌套。

该库的实际做法：维护一个 filter 名称到构造函数的内存表，使用 Array.reduceRight() 从内到外构建装饰器链：

const filterChain = filters.reduceRight(
  (stream, { name, params }) => new FILTER_MAP[name](stream, params),
  rawStream
);

这种函数式组合比 switch-case 更清晰，而且新增解码器只需向 FILTER_MAP 注册即可——开闭原则的教科书级别实现。

十、加密与解密

lib/core/crypto.js + lib/core/decrypt_stream.js

PDF 支持多种加密标准。PDF.js 需要在内置这些解密能力的同时，不能导出加密功能（遵守加密导出法规）。

安全处理器标准加密算法
StandardPDF 1.x - 2.0RC4 (40/128 bit)、AES-128、AES-256
公钥PDF 2.0PKCS#7 信封加密

加密的工作方式：

  1. CipherTransformFactory 从 Trailer 的 Encrypt 字典中提取加密参数（算法、密钥长度、权限标志）
  2. 使用用户提供的密码 + 文件 ID 派生加密密钥（PDF 2.0 的 AES-256 使用 SHA-256 + SHA-384 + SHA-512 的三重哈希）
  3. DecryptStream 包装在解码器链的最内层，透明地在读取时解密数据

关键设计：加密/解密完全集成在流处理管道中——上层代码（Parser、Evaluator）完全不知道数据是否被加密。这保持了各层之间的关注点分离。
