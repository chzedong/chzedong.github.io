一、项目背景

PDF 格式由 Adobe 于 1993 年发布，是电子文档的事实标准。在 PDF.js 出现之前，浏览器渲染 PDF 只能依赖插件（Adobe Reader、Chrome PDF Viewer），这些插件是黑盒的二进制组件，无法被 Web 开发者定制和控制。

PDF.js 的核心使命：用纯 JavaScript 和 HTML5 Canvas 在浏览器中完整解析和渲染 PDF。它不依赖任何原生代码，所有解析逻辑（字体、图像、压缩流、颜色空间）都在 JavaScript 中完成，这使得它天然跨平台、可嵌入任何 Web 应用。

📌 为什么这个库值得深度学习

PDF.js 是一个「全栈 Web 引擎」的绝佳案例：它涵盖了二进制格式解析、状态机驱动的词法分析、流式编解码、字体工程、颜色管理、Canvas 渲染管线、Web Worker 多线程架构，以及一个完整的 UI Viewer。几乎每一个子领域都是独立的技术深水区。

二、快速上手

最小可运行示例

// 1. 引入库
import * as pdfjsLib from "pdfjs-dist";

// 2. 指定 Worker（核心解析在 Worker 线程中执行）
pdfjsLib.GlobalWorkerOptions.workerSrc = "pdf.worker.js";

// 3. 加载 PDF → 获取第一页 → 渲染到 Canvas
const doc = await pdfjsLib.getDocument("/path/to/file.pdf").promise;
const page = await doc.getPage(1);
const canvas = document.createElement("canvas");
await page.render({ canvasContext: canvas.getContext("2d"), viewport: page.getViewport({ scale: 1.5 }) }).promise;
document.body.appendChild(canvas);

常见场景

// 场景1: 从 ArrayBuffer 加载（文件上传）
const doc = await pdfjsLib.getDocument(arrayBuffer).promise;

// 场景2: 流式加载 + Range 请求（大文件按需加载）
const doc = await pdfjsLib.getDocument({
  url: "/large.pdf",
  rangeChunkSize: 65536,     // 每次请求 64KB
  disableAutoFetch: false,   // 自动预取下一页
}).promise;

// 场景3: 获取文本内容
const textContent = await page.getTextContent();
textContent.items.forEach(item => console.log(item.str));
💡 与原生浏览器 PDF Viewer 的等价对比

如果你用过 Chrome 的内置 PDF 查看器，会发现它其实是基于 PDF.js 的 Fork（Chromium 内置的 PDFium 是 C++ 实现）。PDF.js 用 JS 做了同样的事：解析 PDF 二进制 → 生成绘制指令 → Canvas 渲染。区别在于 PDF.js 让你可以完全控制渲染管线的每一环节。

架构设计核心决策：主线程 / Worker 分离

PDF 解析是 CPU 密集型操作（特别是字体解析和图像解码）。PDF.js 将所有解析逻辑放在 Web Worker 中，主线程只负责：

  • 发起 getDocument() 调用
  • 接收 Worker 传回的 OperatorList（图形指令序列）
  • 在 Canvas 上执行绘制
  • 管理 TextLayer、AnnotationLayer 等 DOM 叠加层

Worker 和主线程通过 MessageHandler（基于 postMessage 的 RPC 封装）通信。数据传输使用 Transferable Objects 避免内存复制。

七、历史演进与关键决策

版本关键决策原因

v2.0 (2018)完全重写为 ES6 class + module抛弃旧的 prototype 继承模式，提升可维护性
v2.5支持 XFA 表单（动态 PDF）企业级需求——政府/金融领域的动态 PDF 表单渲染
v5.7 (2026.04)引入 PS Lexer/Parser 的 Wasm 编译器PostScript 函数求值是性能瓶颈，Wasm 加速 10x+
🧠 教学价值最高的设计决策

为什么 OperatorList 而非直接绘制？PDF.js 不直接在 Worker 中绘制（Worker 无法访问 DOM/Canvas），而是生成一个平台无关的图形指令序列（OperatorList）。这个中间表示既可以在浏览器 Canvas 上绘制，也可以导出为 SVG，还可以在 Node.js 中渲染为位图。这是一个教科书级别的「平台抽象层」设计。

八、推荐学习路径

按以下顺序阅读源码，逐步建立对 PDF.js 的心智模型：

  1. 1 lib/shared/util.js 类型定义、OPS 算子表、工具函数——先理解 PDF.js 的「词汇表」
  2. 2 lib/core/primitives.js PDF 基本数据结构：Name、Dict、Ref、Cmd——PDF 世界的「原子」
  3. 3 lib/core/parser.js Lexer + Parser：如何把字节流解析为结构化对象
  4. 4 lib/core/xref.js 交叉引用表：PDF 的对象定位机制
  5. 5 lib/display/api.js 公开 API、Worker 通信——从用户调用到底层执行的桥梁
  6. 6 lib/display/canvas.js CanvasGraphics：2880 行的 Canvas 渲染器
  7. 7 lib/core/evaluator.js PartialEvaluator：4600 行的求值引擎（最复杂模块）
