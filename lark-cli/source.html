💡 设计洞察：入口极简

main.go 只有 17 行。核心逻辑全部封装在 cmd.Execute() 中。 唯一的外部关注点是 extension/credential/env 的空白导入——通过 Go 的 init() 机制自动注册环境变量凭证提供者，无需在 cmd 包中显式引用。

📐 模块划分原则
  • cmd/ + shortcuts/ = 用户可见的命令层
  • internal/ = 不可导出的实现细节，30+ 子包各司其职
  • errs/ = 唯一的公共 API 契约包，外部消费者只依赖它
  • extension/ = Go init() 注册模式实现的可插拔扩展点

Bootstrap 阶段

cmd/bootstrap.go:17-37 
func BootstrapInvocationContext(args []string) (cmdutil.InvocationContext, error) {
    var globals GlobalOptions
    fs := pflag.NewFlagSet("bootstrap", pflag.ContinueOnError)
    fs.ParseErrorsAllowlist.UnknownFlags = true
    RegisterGlobalFlags(fs, &globals)
    if err := fs.Parse(args); err != nil { ... }
    return cmdutil.InvocationContext{Profile: globals.Profile}, nil
}

Bootstrap 阶段在 cobra 命令树构建之前解析全局 flag（如 --profile）， 确保后续配置解析能使用正确的 profile。使用 pflag.ContinueOnError + UnknownFlags=true 的宽容解析策略——只提取已知 flag，忽略子命令参数。

Build 阶段

buildInternal() 是纯组装函数，无任何 IO 副作用。按以下顺序构建命令树：

  1. 创建 Factory — cmdutil.NewDefault() 封装 IO 流、配置、凭证、品牌信息
  2. 创建 rootCmd — 设置 Use/Short/Long/Version
  3. 注册核心子命令 — config, auth, profile, doctor, api, schema, completion, update, event
  4. 注册 API 命令 — service.RegisterServiceCommands() 从 OAPI 元数据生成 100+ 命令
  5. 注册 Shortcuts — shortcuts.RegisterShortcutsWithContext() 挂载 18 个业务域的 +命令
  6. 安装守卫 — installUnknownSubcommandGuard() 为纯分组命令注入错误提示
  7. 安装插件 — installPluginsAndHooks() 加载外部插件并注册生命周期钩子
  8. 策略裁剪 — applyUserPolicyPruning() 根据用户配置裁剪高/中风险命令
🔒 Fail-Closed 安全策略

插件加载失败时采用 Fail-Closed 策略：安装一个 guard stub 命令， 用户执行任何命令都会看到插件错误提示，防止插件安全策略被静默绕过。 YAML 配置错误则采用 Fail-Open（仅警告），避免配置拼写错误导致 CLI 完全不可用。

命令注册机制

Shortcuts 的注册是 lark-cli 最精巧的设计之一。每个业务域包通过 init() 将自己的 shortcuts 聚合到全局切片中：

shortcuts/register.go:61-82（init 函数） 
var allShortcuts []common.Shortcut

func init() {
    allShortcuts = append(allShortcuts, apps.Shortcuts()...)
    allShortcuts = append(allShortcuts, calendar.Shortcuts()...)
    allShortcuts = append(allShortcuts, doc.Shortcuts()...)
    // ... 18 个业务域按需追加
}

RegisterShortcutsWithContext() 按 Service 字段分组， 为每组找到或创建对应的 cobra 命令节点，然后调用 shortcut.MountWithContext() 挂载。 品牌守卫机制 (brandRestrictedServices) 按品牌（飞书/Lark）限制特定域的可用性。

Shortcut 类型与执行流程

common.Shortcut 是一个声明式结构体（shortcuts/common/types.go:38-65）， 定义了命令的完整契约：

  • 元数据：Service, Command, Description, Risk
  • 权限：Scopes, UserScopes, BotScopes（支持按身份区分）
  • 参数：Flags []Flag（声明式 flag 定义）
  • 钩子：Validate, Execute, DryRun, PostMount

错误处理体系

RFC 7807 Problem 设计

lark-cli 的错误体系是它最值得学习的部分之一。它采用 RFC 7807 Problem Details 标准，定义了统一的错误契约（errs/problem.go）： 

type Problem struct {
    Category       Category `json:"type"`           // 顶层分类轴
    Subtype        Subtype  `json:"subtype,omitempty"` // 子类型标识
    Code           int      `json:"code,omitempty"`    // 错误码
    Message        string   `json:"message"`           // 人类可读消息
    Hint           string   `json:"hint,omitempty"`     // 修复建议
    LogID          string   `json:"log_id,omitempty"`   // 日志关联 ID
    Troubleshooter string   `json:"troubleshooter,omitempty"` // 飞书诊断链接
    Retryable      bool     `json:"retryable,omitempty"` // 是否可重试
}
🎯 设计目标

AI Agent 和 shell 脚本可以直接解析结构化 JSON，而不是正则匹配错误字符串。
ProblemOf(err) 提供多态读取；IsXxx(err) 系列断言函数提供类型判断。

TypedError 设计模式

每个 Category 对应一个具体的 *XxxError 类型，嵌入 Problem 并添加类型专属扩展字段。 使用 Builder 模式的链式 API：

errs/types.go:33-38（推荐用法） 
return errs.NewValidationError(errs.SubtypeInvalidArgument,
    "invalid --start: %v", err).
    WithHint("expected RFC3339, e.g. 2026-05-26T10:00:00Z").
    WithParam("--start")
💡 设计亮点
  • Category 由构造函数锁定 — 调用方永远无法误填 Category
  • Subtype + Message 是必选位置参数 — 编译器阻止遗漏
  • WithXxx 返回具体类型指针 — 链式调用不需要类型断言
  • 静态 lint 校验 — lint/errscontract 检查 subtype 是否声明、是否拼写错误
  • nil-receiver safe — 所有 Error/Unwrap 方法对 nil 指针安全

配置与认证

多应用配置模型

MultiAppConfig（internal/core/config.go:61-66）支持在一个 CLI 实例中管理多个飞书应用。 每个应用有独立的 AppID、AppSecret、品牌、语言和权限配置： 

type MultiAppConfig struct {
    StrictMode  StrictMode  `json:"strictMode,omitempty"`
    CurrentApp  string      `json:"currentApp,omitempty"`   // 当前活跃应用
    PreviousApp string      `json:"previousApp,omitempty"`  // 上一个应用（支持 quick-swap）
    Apps        []AppConfig  `json:"apps"`
}

应用选择优先级：--profile flag → CurrentApp 字段 → Apps[0]

品牌适配

通过 LarkBrand 枚举（internal/core/types.go:9-12）区分飞书（中国）和 Lark（国际）： 

const (
    BrandFeishu LarkBrand = "feishu"  // open.feishu.cn
    BrandLark   LarkBrand = "lark"    // open.larksuite.com
)

ResolveEndpoints() 根据品牌返回不同的 API 端点 URL。品牌守卫在 Shortcuts 注册时生效， 例如 apps 域仅在飞书品牌可用。

凭证管理

凭证存储通过 internal/keychain 接口抽象，支持 OS 原生密钥链。 extension/credential/env 通过 init() 注册环境变量凭证提供者作为回退方案。

Secret 字段使用 SecretInput 类型，支持加密存储和延迟解析。 Workspace 管理（internal/core/workspace.go）处理多 profile 的隔离。

关键设计模式

模式应用位置说明

Chain of Responsibility cmd/root.go → handleRootError 多级错误分发：TypedError → 旧版 ExitError → Cobra 错误，逐级匹配

学习要点总结

工程化启示

  • 入口极简 — main.go 只有 17 行，所有逻辑委托给 cmd 包
  • Build/Execute 分离 — buildInternal() 纯组装无副作用；Execute() 处理运行时行为
  • Fail-Closed 安全策略 — 插件加载失败时阻止命令执行，而非静默降级

架构设计启示

  • 声明式 Shortcut — 用结构体字段声明命令行为，框架自动生成 cobra 命令，大幅减少样板代码
  • RFC 7807 错误契约 — 统一的结构化错误，AI Agent 可直接解析
  • TypedError + Builder 链 — Category 构造函数锁定 + 链式 WithXxx，编译器保证正确性
  • init() 注册器模式 — 新增业务域只需添加文件，无需修改中央注册表

可复用技巧

  • vfs.FS 接口 — 文件系统抽象让测试不依赖真实文件
  • BuildOption 函数选项 — Golang 惯用的可选配置模式
  • 品牌守卫 map — 简单的 map[string][]Brand 实现多品牌差异化
  • lint/errscontract — 自研 lint 工具保证错误 subtype 的一致性
