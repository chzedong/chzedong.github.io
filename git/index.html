1. 项目背景

Git 由 Linus Torvalds 于 2005 年 4 月创建，初衷是为 Linux 内核开发提供一个高性能、分布式、不可篡改的版本控制系统。 在 BitKeeper 许可证争端之后，Linus 花了约 10 天 写出了 Git 的第一个可用版本。

💡 设计目标（Linus 原话）
  • 将 Concurrent Versions System (CVS) 作为反面教材 — 所有 CVS 做错的事情，Git 都要做对
  • 支持分布式工作流 — 不依赖中央服务器
  • 强数据完整性 — 用 SHA-1 哈希保证内容不可篡改
  • 高性能 — 分支切换、合并、diff 操作均为 O(1) 或接近 O(1)

核心解决的问题：

  • 数据完整性：每个对象由 SHA-1 哈希寻址，任何篡改都会改变哈希值
  • 分布式协作：每个 clone 都是完整仓库副本，无需网络即可提交、分支、合并
  • 非线性开发：轻量级分支（一个 41 字节的文件）鼓励并行开发
  • 高效存储：通过 delta 压缩和 packfile 大幅减少存储开销

1.5 快速上手

最小可运行示例

# 初始化仓库并创建第一个提交
git init my-project && cd my-project
echo "hello git" > README.md
git add README.md && git commit -m "first commit"

查看 Git 内部对象的底层命令

# 查看 commit 对象内容
git cat-file -p HEAD

# 查看 tree 对象内容
git cat-file -p HEAD^{tree}

# 查看 blob 对象的原始内容
git hash-object README.md          # 计算 hash
git cat-file -p <blob_hash>       # 查看内容

常见协同场景

# 场景1：分支 + 合并
git checkout -b feature/x
# ... 修改文件 ...
git commit -am "feature done"
git checkout main
git merge feature/x

# 场景2：协作冲突解决
git fetch origin
git rebase origin/main        # 变基到最新，可能产生冲突
# ... 解决冲突后 ...
git add . && git rebase --continue

2. 数据原理 — Git 对象模型

Git 的核心本质上是一个内容寻址文件系统 (content-addressable filesystem)， 上层再包装了一套 VCS 用户界面。理解了四种对象类型，就理解了 Git 的全部数据流。

    subgraph 暂存区
        IDX["index / staging area\n(.git/index)"]
    end
    subgraph 对象数据库
        B[Blob\n文件内容]
        T[Tree\n目录结构]

    IDX -- "git commit" --> C
    C --> T
    T --> B
    C --> C
    BR --> C
    HD --> BR
    TG --> C

2.1 Blob — 内容寻址的基石

Blob (Binary Large Object) 只存储文件内容，不存储文件名、路径或权限。 同一个内容只有一个 blob，所有文件名指向它。这是 Git 去重的基础。 

# Blob 对象格式（Git 内部存储）
"blob <content_length>\0" + file_content
# SHA-1("blob 11\0hello world") = 3b18e512dba79e4c8300dd08aeb37f8e728b8dad
🧠 如果让你来实现...

问题：如何存储文件的不同版本？

直觉方案：以文件路径为 key，存储增量差异（类似 SVN）

Git 的实际做法：以内容的 SHA-1 哈希为 key，相同内容只存一份。文件名/路径存储在 tree 对象中，与内容数据分离

差异分析：直觉方案使得重命名/移动文件的 diff 变得昂贵；Git 的方案天然支持去重、内容校验，且文件移动只改变 tree 而不改变 blob

2.2 Tree — 目录快照

Tree 对象 存储目录结构：每一行是一个条目，包含文件模式、对象类型（blob 或 tree）、SHA-1 哈希和文件名。 Tree 递归嵌套 Tree，形成完整的目录树。 

# Tree 条目格式（简化）
100644 blob a906cb2a4a904a152e80877d4088654daad0c859  README.md
100755 blob 8f7dfe28c5c5e7a1b4d2a95c2f6d2c4f8e1a3b6d  script.sh
040000 tree 4b825dc642cb6eb9a060e54bf8dc4000fb5c0f5a  src

2.3 Commit — 不可变历史节点

Commit 对象 是 Git 历史的基本单元，包含：一棵 tree（项目快照）、 零到多个 parent commit（形成 DAG）、作者/提交者信息、时间戳和提交信息。 

# git cat-file -p HEAD 的典型输出
tree 4b825dc642cb6eb9a060e54bf8dc4000fb5c0f5a
parent a1b2c3d4e5f6789012345678901234567890abcd
author Linus Torvalds <torvalds@linux-foundation.org> 1716500000 -0700
committer Linus Torvalds <torvalds@linux-foundation.org> 1716500000 -0700

Add feature X

This commit introduces the long-awaited feature X.
Signed-off-by: ...
💡 关键洞察：commit 不可变

一旦创建，commit 对象 永不修改。修改历史（如 rebase、amend）实际是创建新 commit，旧 commit 保留在对象库中直到被 GC 回收。这就是为什么 git reflog 能恢复 "丢失" 的提交。

2.4 Tag — 带注释的里程碑

轻量标签 (Lightweight Tag) 只是一个指向 commit 的引用（ref）， 而 注释标签 (Annotated Tag) 本身是一个 Git 对象，包含标签名、信息、日期和 GPG 签名。 

# Annotated tag 对象内容

2.5 引用层级 — 命名的指针

Git 的所有"命名" —— 分支、标签、HEAD、远程引用 —— 都只是指向某个 SHA-1 的指针文件。

# .git/refs/heads/main 的内容（仅41字节）
4b825dc642cb6eb9a060e54bf8dc4000fb5c0f5a

# HEAD 是一个符号引用
# .git/HEAD 的内容：
ref: refs/heads/main                  # 正常状态 → 指向一个分支
4b825dc642cb6eb9a060e54bf8dc4000fb5c0f5a  # detached HEAD → 直接指向 commit

2.6 Packfile — 存储压缩

Git 默认以 loose object 格式存储每个对象（zlib 压缩后放入 .git/objects/xx/xxxx...）。 当对象过多或执行 git gc 时，会将它们打包成 packfile， 在其中通过 delta 差异链只存储对象间的增量。 

flowchart LR
    subgraph Loose
        O1["obj/3b\n/18e..."]
        O2["obj/a9\n/06c..."]
        O3["obj/8f\n/7df..."]
    end
    subgraph Packed
        P["pack-xxx.pack\n最新版(全量) ← delta1 ← delta2"]
        IDX["pack-xxx.idx\n哈希 → 偏移量 索引"]
    end
    Loose -- "git gc" --> Packed

图：Loose object → Packfile 的压缩流程

⚠️ 性能注意

Packfile 按启发式算法选择 delta base —— 优先选择最近的文件版本、相同命名的文件、大小相近的文件。这是 Git 存储效率高的核心原因，但也是 git gc 耗时的主要来源。

3. 暂存区 (Index) 原理

暂存区是 Git 区别于其他 VCS 的核心设计。.git/index 是一个二进制文件，存储了下次提交的完整 tree 快照。 它不是一个 "diff"，而是一个完整的待提交视图。 

flowchart TD
    WD["工作区\n(file stats)"]
    IDX["暂存区 Index\n(.git/index)"]
    OBJ["对象数据库\n(.git/objects)"]

    WD -- "git add → 计算 SHA-1\n将 blob 写入 objects\n更新 index 条目" --> IDX
    IDX -- "git commit → 将 index 的内容\n写入 tree + commit 对象" --> OBJ
    OBJ -- "git checkout → 将 tree\n展开到 index 和工作区" --> IDX
    OBJ -- "git checkout → 将 tree\n展开到 index 和工作区" --> WD

图：Index 在三个区域间的桥梁角色

Index 条目结构（简化）

// .git/index 中的每条记录
ctime_sec, ctime_nsec     // 文件元数据变更时间
mtime_sec, mtime_nsec     // 文件内容修改时间
dev, ino, mode, uid, gid  // stat 信息（用于快速变更检测）
file_size
sha1                      // blob 的 SHA-1 哈希
flags                     // assume-unchanged, intent-to-add 等
file_path                 // 以 null 结尾的路径

问题：如何在提交前预览将要提交的内容？

直觉方案：提交时计算工作区与 HEAD 的 diff，选择性地包含某些文件

Git 的实际做法：Index 是一个完整的中间快照，git add 就是在构建这个快照

差异分析：直觉方案在部分提交时计算复杂（"我只提交这 3 行"），而 Index 方案允许逐块暂存（git add -p）， 因为它操作的是一个明确的中间状态而非动态 diff

4. 核心协同策略

4.1 分布式协同模型

Git 是完全分布式的版本控制系统。没有 "中央仓库" 的架构概念 —— 每个 clone 拥有完整的历史、所有分支和标签。这是通过 Git 的对象模型实现的： 只要你有 commit 的 SHA-1，就能从对象数据库中取出完整的 tree 和 blob。

    subgraph A["开发者 A 仓库"]
        A_MAIN["main"]
        A_FEAT["feature/a"]
    end
    subgraph B["开发者 B 仓库"]
        B_MAIN["main"]
        B_FIX["fix/bug"]
    end
    subgraph R["远程仓库 origin"]
        R_MAIN["origin/main"]
        R_DEV["origin/dev"]
    end

    A -- "git push" --> R
    R -- "git fetch/pull" --> A
    B -- "git push" --> R
    R -- "git fetch/pull" --> B
    A <-..->|"format-patch / email"| B

图：分布式模型 — 远程仓库只是另一个 Git 仓库；协作者也可以直接交换 patch

4.2 Fetch / Pull / Push 的数据流

  1. git fetch：从远程下载缺失的 object（blob/tree/commit）， 更新 refs/remotes/origin/*，但不修改本地分支和工作区
  2. git merge / git rebase：将远程跟踪分支的变更整合到本地分支
  3. git pull = git fetch + git merge/rebase：两步合一
  4. git push：将本地 commit 发送到远程，要求远程分支是快进 (fast-forward) —— 非快进推送会被拒绝，防止覆盖他人工作

4.3 Rebase vs Merge — 两种整合策略

维度MergeRebase

冲突解决 一次性解决所有冲突 每个 commit 可能独立产生冲突

4.4 远程跟踪分支与 reflog

远程跟踪分支 (refs/remotes/origin/main) 是上次 fetch 时远程仓库状态的本地缓存。 它不是你本地的 main 分支 —— 你可以在本地 main 上继续提交， 然后用 git diff origin/main 对比差异。

Reflog (.git/logs/) 记录了每个引用在本地的变更历史（默认保留 90 天）， 是 "救命稻草"：误操作 git reset --hard 后可通过 git reflog 找回丢失的 commit。

5. 冲突解决方案 — 剖析三方合并

冲突的本质：两个修改者对同一段代码做出了不同的修改，且无法自动判定哪个是正确的。 理解 Git 如何 检测、标记、帮助解决 冲突，是掌握协同开发的关键。

5.1 三方合并 (Three-Way Merge) 算法

flowchart TD
    BASE["Base (共同祖先)\nline 1\nline 2\nline 3"] --> OUR
    BASE --> THEIR
    OUR["Ours (当前分支)\nline 1\nline 2 modified by us\nline 3"] --> RESULT
    THEIR["Theirs (合并分支)\nline 1\nline 2 modified by them\nline 3"] --> RESULT
    RESULT{"三方比对"}
    RESULT -->|"只有一边修改"| AUTO["自动采用修改"]
    RESULT -->|"两边都修改"| CONFLICT["标记为冲突"]
    RESULT -->|"都没修改"| UNCHANGED["保持原样"]

图：三方合并的决策树 — 以共同祖先 (merge-base) 为裁判

Merge-base 的计算：Git 使用 git merge-base 找到两个分支的最佳共同祖先 (Best Common Ancestor)。 在有多个共同祖先时（交叉合并场景），recursive 策略会先合并共同祖先，再将结果作为虚拟 base。

5.2 冲突标记解读

<<<<<<< HEAD                  # 当前分支 (ours) 的内容开始
function greet(name) {
  console.log("Hello, " + name);
}
=======                        # ours / theirs 分隔符
function greet(name, lang) {
  console.log(`Hello, ${name}`);
}
>>>>>>> feature/i18n        # 合并分支 (theirs) 的内容结束
⚠️ 常见误区

误区：<<<<<<< 标注的是 "修改前"，>>>>>>> 是 "修改后"

实际：HEAD → 你当前分支的内容；feature/i18n → 你正在合并的分支的内容。与时间先后无关，只取决于你的当前 checkout 和合并方向。

5.3 Rerere — 重用已解决的冲突

git rerere (Reuse Recorded Resolution) 是 Git 中最被低估的功能之一。 它记录你如何解决一个冲突，并在下次遇到相同冲突时自动应用之前的解决方案。 

# 启用 rerere
git config --global rerere.enabled true

# 工作流程：
# 1. 合并产生冲突 → 手动解决 → git add → git commit
#    Git 自动在 .git/rr-cache/ 中记录此次解决方案
# 2. 下次遇到相同冲突时：
#    git merge feature/x  →  Git 自动应用之前的解决方案
sequenceDiagram
    participant U as 开发者
    participant G as Git
    participant C as rr-cache/

    U->>G: git merge feature/a
    G->>G: 检测冲突
    G->>C: 查询是否有已记录解决方案
    C-->>G: 无记录
    G->>U: 标记冲突（<<<<<<< ... >>>>>>>）
    U->>G: 手动解决 → git add → git commit
    G->>C: 记录 preimage + postimage
    Note over C: 冲突解决模式已保存

    U->>G: git merge feature/b (相同冲突)
    G->>C: 查询命中
    C-->>G: 返回已记录的解决方案
    G->>U: 自动应用解决方案，无人工干预

图：Rerere 在一个长生命周期分支反复合并时的价值

策略算法适用场景冲突处理

递归三方合并的现代重写		 所有普通合并 逐文件三方比对，支持 rename 检测

6. 算法与数据结构深度剖析

6.1 提交历史 — 有向无环图 (DAG)

Git 的提交历史是一个 DAG：每个 commit 是一或多个 parent commit 的子节点， 由于 SHA-1 的不可逆性和不可变性，图中不可能出现循环。 

flowchart TD
    C1["C1 (root)"] --> C2["C2"]
    C2 --> C3["C3 (merge)"]
    C2 --> C4["C4"]
    C4 --> C3
    C3 --> C5["C5 (HEAD)"]
    C3 --> C6["C6 (feature)"]
    C5 --> C7["C7"]
    C6 --> C7

图：Git 的提交图是 DAG — C7 是一个 merge commit，有两个 parent

关键操作复杂度：

操作复杂度说明
新增 commitO(1)只创建新节点并更新 HEAD 引用
切换分支O(n)n = 工作区文件数，需更新 index + 文件系统
git logO(k)k = 遍历的 commit 数，沿 parent 链遍历
merge-baseO(N)N = 历史深度，BFS/双向 BFS 找到最佳共同祖先
git blameO(N)N = 文件的历史 commit 数

6.2 Diff 算法 — Myers 差分算法

Git 使用 Myers Diff Algorithm 计算两个文本之间的最小编辑距离 (edit script)， 生成人类可读的 diff 输出。这是 Git 显示变更的基础。 

flowchart TD
    SRC["文件 A\n(旧版本)"] --> EDIT["Myers 最短编辑脚本\n(Shortest Edit Script)"]
    DST["文件 B\n(新版本)"] --> EDIT
    EDIT --> DIFF["统一 diff 格式输出\n- 删除行 (红色)\n+ 新增行 (绿色)"]

图：Diff 算法输入 → 编辑脚本 → 输出

Myers 算法核心思想：

  1. 将两个文本的比对问题建模为一张 编辑图 (edit graph)：横轴走旧文本，纵轴走新文本
  2. 对角线移动 = 匹配字符（免费），水平/垂直移动 = 删除/插入（代价为 1）
  3. BFS 遍历编辑图，找到从 (0,0) 到 (M,N) 的最短路径
  4. 时间复杂度 O((M+N)·D)，其中 D = 编辑距离；空间复杂度 O((M+N)·D)
🧠 如果让你来实现...

问题：如何找到两个文件的差异？

直觉方案：逐行比对，用 LCS (最长公共子序列) DP 算法

Git 的实际做法：Myers 差分算法 — 编辑图上的 BFS 最短路径搜索

差异分析：LCS DP 在所有情况下都是 O(N²) 时空复杂度，但实际 diff 场景中两个文件通常很相似（D ≪ N）。 Myers 算法利用了这一特性，复杂度与差异大小成正比，在代码 diff 场景下远优于 LCS DP

6.3 Delta 压缩 — XDelta 差异链

Git 在 packfile 中使用 delta 压缩 来减少存储：一个对象可以表示为 "以另一个对象为基础，加上这个 patch"。 Git 使用 XDelta 格式存储差异链，同时通过启发式算法选择 delta base。 

# Delta 对象结构（简化）
source_object_sha1           # 基准对象的 SHA-1
target_object_size           # 解压后的大小
delta_instructions:          # 增量指令序列
  COPY  offset=0   len=50  # 从 base 复制 50 字节
  INSERT data="new content"  # 插入新内容
  COPY  offset=60  len=200 # 从 base 跳过 10 字节后复制 200 字节

Delta base 选择启发式（按优先级）：

  1. 同名文件的历史版本（最优先）
  2. 同目录下的其他文件
  3. 文件大小相近的 blob
  4. 最新版本优先作为 base（最近被访问概率最高）

6.4 垃圾回收 (git gc)

git gc 负责清理不可达对象、打包 loose object、压缩 packfile。 其核心流程：

  1. 可达性分析：从所有引用（分支、标签、HEAD、reflog）出发，标记所有可达对象
  2. 打包 loose object：将松散对象打包为 packfile，应用 delta 压缩
  3. 清理不可达对象：不可达且超过保护期（默认 2 周）的对象被删除
  4. 合并 packfile：将多个小 packfile 合并为一个大 packfile（默认目标：单个 pack）
  5. 清理 reflog 过期条目：超过 90 天的 reflog 记录被移除
🛑 注意

git gc --prune=now 会立即删除所有不可达对象，恢复期为零。 这在你确认不需要恢复任何数据时才使用。日常使用 git gc --auto（提交后自动触发）是更安全的选择。

💡 Git vs Mercurial

Git 和 Mercurial 都是分布式 VCS，数据模型相似。Git 胜在更高的灵活性（index 设计、更丰富的操作原语）、更大的生态（GitHub）。 Mercurial 胜在更一致的用户界面和内建的扩展机制。但 Git 的 "底层 plumbing + 上层 porcelain" 双层设计使其更适合脚本化和自动化。

8. 架构设计深度剖析

8.1 Plumbing vs Porcelain 双层架构

Git 被刻意设计为两层：底层 plumbing（管道）提供原子操作原语； 上层 porcelain（瓷器）提供用户友好的高级命令。这使得 Git 可以像 UNIX 管道一样组合。

层级目标用户示例命令
Porcelain (瓷器)日常开发者git commit, git merge, git rebase
Plumbing (管道)脚本 / 工具开发者git hash-object, git cat-file, git update-index
🧠 如果让你来实现...

问题：如何设计版本控制系统的命令体系？

直觉方案：为每个用户场景提供一个命令（checkout, commit, merge），内部实现对用户不可见

Git 的实际做法：将底层原语暴露为 plumbing 命令，porcelain 命令是它们的组合。用户可以用 plumbing 构建自定义工作流

差异分析：直觉方案导致 "要么用我的 GUI/CLI，要么重新造轮子"。 Git 的方案让 GitHub、GitLab、IDE 插件等可以基于 plumbing 构建上层工具，而不用碰 Git 内部 C 代码。 这是 Git 生态如此庞大的核心原因

8.2 设计模式

  • 内容寻址存储 (Content-Addressable Storage)：对象由内容哈希寻址，天然去重 + 完整性校验
  • 不可变数据结构 (Immutable Data)：所有对象创建后不可修改，"修改" = 创建新对象
  • 写时复制 (Copy-on-Write)：分支 / commit 共享不变的 tree/blob，只需存储差异
  • 引用透明：SHA-1 引用完全透明，任何仓库可以通过 SHA-1 交换对象

9. 历史演进 & 关键决策

版本决策为什么
v0.99 (2005) 选择 SHA-1 作为哈希算法 SHA-1 在当时是最广泛使用的安全哈希，Git 对碰撞的容忍度较高（内容寻址而非安全签名）
v1.5.0 (2007) 引入 git add -p 交互式暂存 Index 设计使得逐块暂存成为可能，这是 SVN 等增量模型无法优雅实现的特性
v1.7.12 (2012) 引入 rerere 自动冲突解决 内核开发中长生命周期分支反复合并的痛点：相同冲突不应让人重复解决
v2.19 (2018) 引入 range-diff 解决 rebase 后难以对比 "修改了什么" 的问题 — 对比两个 commit 范围的差异
v2.34 (2021) ort 替代 recursive 作为默认合并策略 ort 是 recursive 的完整重写，性能提升显著（特别是 rename 检测），且代码更可维护（~3000 行 vs ~5000 行）
v2.45 (2024) SHA-256 兼容性基础设施 应对 SHA-1 碰撞攻击（如 SHAttered），Git 逐步迁移到 SHA-256，同时保持与 SHA-1 仓库的互操作性

10. 学习要点总结

核心概念记忆清单

  1. Git 本质是内容寻址文件系统：SHA-1 → Object → Content，版本控制是上层语义
  2. 四种对象：Blob (内容) → Tree (目录) → Commit (快照) → Tag (标签)，层层嵌套
  3. 引用只是指针：分支 = 41 字节的文件，HEAD = "ref: refs/heads/xxx"
  4. Index 是完整快照：不是 diff，是你下次 commit 将要创建的那个 tree
  5. 三方合并是冲突解决的基础：BASE + OURS + THEIRS → 自动/冲突/不变
  6. Rebase = 重写历史：不改变内容，只改变 parent 链，切勿 rebase 已推送的提交
  7. Myers 算法利用 D ≪ N 特性：在代码 diff 场景大幅优于 LCS DP
  8. Plumbing 设计是生态基石：GitHub/GitLab/IDE 插件基于 plumbing 而非内部 API
  9. Reflog 是后悔药：90 天内的误操作几乎都能恢复
  10. Packfile + Delta 压缩：同名文件、大小相近、最新优先 —— 这是 Git 存储高效的核心
