Vitest — 下一代测试框架

由 Vite 驱动的极速单元测试框架。原生 ESM、智能 Watch 模式、Jest 兼容 API，重新定义前端测试开发体验。

v5.0.0-beta.3 MIT License TypeScript pnpm Monorepo Vite-Powered VoidZero

1. 项目背景

Vitest 由 Anthony Fu 和 VoidZero 团队于 2021 年创建，目标是解决前端测试领域的一个长期痛点：测试框架与构建工具的割裂。Jest 虽然生态成熟，但其基于 jest-runtime 的模块系统需要额外配置才能理解 TypeScript、ESM、JSX 等现代语法——而这些恰好是 Vite 天然支持的。

Vitest 的核心洞察：复用 Vite 的转换管道（transform pipeline）、解析器（resolver）和插件系统，零配置即可获得与开发环境一致的测试运行环境。这不仅消除了配置负担，更重要的是确保了测试行为与应用运行行为完全一致。

🪄 设计哲学

一个配置，两处运行。 你的 vite.config.ts 既用于开发/构建，也直接驱动测试——不需要 jest.config.js、babel.config.js 等叠加配置。

解决的问题：

  • Jest 配置繁琐（TypeScript/ESM/CSS Modules 需要额外 transformer）
  • Jest 启动慢（vitest 的 watch 模式利用 Vite 的 HMR 机制实现毫秒级重跑）
  • 测试环境与开发环境不一致（不同的模块解析、不同的转换规则）
  • 缺乏原生 ESM 支持（vitest 原生支持 top-level await、import.meta）

2. 快速上手

最小可运行示例——3 行代码开始测试：

sum.test.ts 零配置，直接运行 
import { expect, test } from 'vitest'
import { sum } from './sum'

test('adds 1 + 2 to equal 3', () => {
  expect(sum(1, 2)).toBe(3)
})

常见场景：

带 Mock 的测试
import { vi, test, expect } from 'vitest'
import { fetchUser } from './api'

// Mock 一个模块
vi.mock('./api', () => ({
  fetchUser: vi.fn(() => Promise.resolve({ name: 'Alice' }))
}))

test('mock example', async () => {
  const user = await fetchUser(1)
  expect(user.name).toBe('Alice')
  expect(fetchUser).toHaveBeenCalledWith(1)
})
快照测试
import { expect, test } from 'vitest'

test('snapshot', () => {
  const data = { id: 1, items: ['a', 'b', 'c'] }
  expect(data).toMatchSnapshot()
})

值得学习的工程化实践：

  • Catalog 模式：pnpm-workspace.yaml 中用 catalog 字段统一管理所有依赖版本，避免版本漂移
  • Patch 管理：对上游依赖的修改通过 patches/ 目录 + pnpm 的 patchedDependencies 机制管理
  • 自举测试：test/ 目录下的集成测试使用 vitest 本身运行，保证发布质量
  • 多构建产物：每个子包独立构建，通过 exports 字段细粒度控制导出

6. 核心链路概览

6.1 一次完整测试运行的生命周期

  1. CLI 启动
    packages/vitest/src/node/cli.ts → 使用 cac 解析命令行参数 → 创建 Vitest 实例
  2. 配置解析
    packages/vitest/src/node/core.ts:252 → resolveConfig() 合并 CLI 选项 + vite.config.ts + vitest 默认值
  3. Vite Server 创建
    复用或创建 ViteDevServer → 注册 vitest 插件 → 配置模块解析
  4. 项目发现
    resolveProjects() → 扫描 workspace 中的 test projects → 创建 TestProject 实例
  5. 文件收集
    VitestSpecifications → 根据 include/exclude 模式找到测试文件 → AST 分析提取测试元数据
  6. Worker 池分配
    ProcessPool → 根据 pool 配置（threads/forks/vmThreads）创建 worker 进程
  7. 模块加载与转换
    Worker 内通过 Vite 的 SSR 模块图 加载测试文件 → Vite 插件管道转换 TS/JSX/CSS
  8. 测试执行
    @vitest/runner → startTests() → 按序执行 suite → beforeEach → test → afterEach
  9. 结果报告
    Reporter 管道 → 终端输出 / JSON / JUnit / HTML 报告

🧠 设计取舍

为什么基于 Vite 而不是自己实现模块加载？

复用 Vite 的转换管道意味着 Vitest 天然支持 Vite 生态中的所有插件和配置。这不是一个"偷懒"的决定，而是一个架构级的判断：测试框架不需要也不应该重新发明模块加载。这不仅减少了维护负担，更重要的是保证了测试环境与打包环境的行为一致性。

💡 一句话总结

Vitest 不是"更好的 Jest"——它是利用 Vite 生态系统重新思考测试框架的产物。它的最大创新不是某个具体功能，而是证明了 构建工具和测试框架可以共享同一条管道，从而消除了前端开发中"配置地狱"的最后一公里。
