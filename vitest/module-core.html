⚙️ Vitest 核心架构

packages/vitest 主包 — CLI 入口、配置解析、Worker 池管理、Vite 集成、Reporter 管道。

1. CLI 入口

Vitest 的 CLI 非常精简：仅 3 行代码。它使用 cac（一个迷你 CLI 框架）管理命令。入口文件委托给 ./cli/cac 创建 CLI 实例。

packages/vitest/src/node/cli.ts
import { createCLI } from './cli/cac'
createCLI().parse()

2. 配置解析

resolveConfig() 是 Vitest 的核心配置合并函数，位于 packages/vitest/src/node/config/resolveConfig.ts。它按优先级合并三部分配置：

  1. vitestDefaults（内置默认值）
  2. vite.config.ts 中的 test 字段
  3. CLI 参数（最高优先级）
💡 关键设计

Vitest 复用 Vite 的 defineConfig 和 mergeConfig，在 vite.config.ts 中通过 test 字段配置测试行为。这意味着 一个文件同时定义构建和测试配置——不需要维护 jest.config.js 或 vitest.config.ts（虽然 vitest 也支持独立配置文件）。

3. Worker 池管理

Worker 池负责创建和管理测试执行的工作进程/线程。支持三种池模式：

模式类型适用场景
threadsWorker Threads默认，轻量，共享内存
forksChild Process需要进程隔离（原生模块）
vmThreadsVM + Threads极致隔离（实验性）

Worker 池使用 birpc（双向 RPC 库）与主进程通信。主进程通过 RPC 发送 startTests() 命令，Worker 返回测试结果。

packages/vitest/src/node/pool.ts
// ★ ProcessPool: 管理 Worker 生命周期
export interface ProcessPool {
  runTests(specs: TestSpecification[]): Promise<void>
  close(): Promise<void>
}

4. Multi-Project 系统

Vitest 支持在同一个 workspace 中运行多个项目（monorepo 场景）。每个 TestProject 实例拥有独立的 ViteDevServer、配置和测试文件列表。

packages/vitest/src/node/core.ts:336-349
// ★ 项目解析：扫描 workspace 中的测试项目
this.projects = await this.resolveProjects(this._cliOptions)
if (this._cliOptions.benchmarkOnly) {
  this.projects = this.projects.filter(c => c.config.benchmark.enabled)
}
// ★ 触发 configureVitest 插件钩子
await Promise.all(this.projects.flatMap((project) => {
  const hooks = project.vite.config.getSortedPluginHooks('configureVitest')
  return hooks.map(hook => hook({ project, vitest: this }))
}))

5. Reporter 管道

Vitest 的 Reporter 系统采用观察者模式——多个 Reporter 同时监听测试事件，各自产生不同格式的输出：

Reporter输出
default终端彩色输出
verbose详细模式（每个测试结果）
jsonJSON 文件（机器可读）
junitJUnit XML（CI 集成）
htmlHTML 报告
blob二进制 blob（跨 worker 合并）

7. "如果我来写" 对比思考

🧠 问题：如何管理 Vite Server 的重启？

直觉方案：在 watch 模式下监听文件变更，手动重启 server。

Vitest 的做法：劫持（hijack）server.restart 方法（core.ts:308-314）。在 server 重启前，Vitest 先执行 onRestart 监听器、上报事件、关闭所有 pool 和 project。

差异分析：劫持模式避免了"文件变更 → 手动关闭 → 重建 → 重跑"的脆弱链。通过插件化重启钩子，Vitest 保证 重启是可扩展的——第三方插件可以注册 onRestart 回调来清理自己的状态。

🧠 问题：如何处理多个 Project 的配置隔离？

直觉方案：所有 Project 共享一个全局 ViteDevServer。

Vitest 的做法：每个 TestProject 持有独立的 ViteDevServer 环境（通过 server.environments）。这意味着不同 Project 可以使用不同的模块解析、不同的测试环境（jsdom/happy-dom/node）。

差异分析：共享 server 虽然简单，但不同 project 可能有冲突的配置（比如不同的 resolve.alias）。独立环境带来的 配置隔离性 是 monorepo 场景正确运行的基础。
